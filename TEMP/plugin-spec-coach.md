# Техническое задание: Плагин "Стратегическая коуч-сессия"

## Роль и бизнес-ценность
Инструмент для проведения глубоких стратегических сессий (обзор проектов, распаковка смыслов, планирование, поиск решений, сюжетные линии), который совмещает интерактивный коучинговый чат и динамическое структурирование результатов на древовидном холсте. 

## Основной интерфейс (Двойной вид)
Сущность плагина опирается на разделенный экран:
1.  **Левая боковая панель (Левая колонка)**: Чат для общения с ИИ-коучем.
2.  **Основной экран (Холст)**: Привычное дерево PuuNote (`BoardView`), где в реальном времени строится и фиксируется артефакт сессии.

## Логика работы (Workflows)

### 1. Чат (Модуль Коуча)
*   **ИИ-Квестор**: Задается скрипт-промпт (выбранный фреймворк или роль), по которому ИИ ведет собеседника.
*   **Взаимодействие**: Пользователь отвечает в окне чата. Вывод ИИ в формате текстовых сообщений.
*   *Опционально:* Интеграция с другими плагинами "Голосовой ввод" и "Генерация текста-в-голос (TTS)" для полноценного Voice-to-Voice взаимодействия.

### 2. Фоновое структурирование (Модуль Архитектора)
*   Параллельно диалогу в чате запускается второй агент-аналитик.
*   **Триггер**: Срабатывает после каждой (или заданного количества) итераций вопрос-ответ.
*   **Входные данные**: 
    1. Исходная история последних сообщений (Транскрипция/Чат).
    2. Текущее абстрактное представление уже имеющегося дерева PuuNote (карта-артефакт).
*   **Действие**: Агент-Архитектор использует инструмент (structured output / function calling / JSON), генерирующий инструкции мутации для текущей карты. Например: "создать новую ветку решений", "уточнить пункт Х деталью из ответа".
*   **Результат**: Левая панель продолжает диалог, правая — обрастает карточками.

## Технические требования
1.  У плагина будет кастомный компонент `sidebarComponent`, который рендерит чат.
2.  Плагин должен уметь сохранять историю диалога (чтобы сессию можно было продолжить). Возможно, история диалога будет привязана к корневой ноде текущего документа в виде метаданных `pluginData`, либо сохраняться в отдельном файле (если будет слой файловой системы для проекта).
3.  Для "Модуля Архитектора" потребуется вызывать API `documentApi.addChild`, `addSibling`, `updateContent` программно (без участия пользователя), используя `PluginAPI`.
4.  Возможно, потребуется предоставить плагину возможность читать "структуру всего дерева" для передачи Архитектору в качестве контекста.

## План разработки
1.  **Фаза 1 (Песочница)**: Подготовка ядра: реализация левого SidePanel, расширение PluginAPI.
2.  **Фаза 2 (UI Чата)**: Интеграция простого чата (без автоматического экспорта дерева) для проверки коуч-промптинга.
3.  **Фаза 3 (Интеграция Дерева)**: Разработка агента-архитектора и его промпта. Подключение "тихого" обновления карточек через `PluginAPI.dispatch` при получении структурированного ответа от ИИ.
4.  **Фаза 4 (Голос)**: Добавление Text-to-Speech (воспроизведение текста) в качестве независимого плагина и возможность интеграции/компоновки этих плагинов.
