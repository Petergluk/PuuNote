## 1. Что такое PuuNote?

**PuuNote** — это local-first редактор нелинейных Markdown-документов. В отличие от обычных редакторов (Google Docs, Notion), документ здесь хранится не как один длинный текст, а как **дерево карточек (PuuNode[])**. Корневые идеи — слева, дочерние уточнения раскрываются вправо колонками.

Проект написан целиком на клиенте (браузер), данные хранятся локально в IndexedDB. Сервер не нужен. Есть экспериментальная AI-функциональность (голосовой ввод через Gemini, заглушка генерации карточек) и основа плагинной системы.

### Как это работает

1. **Данные**: документы хранятся в IndexedDB (Dexie). Массив `PuuNode[]` — плоский, дерево строится на лету функцией `buildTreeIndex()`.
2. **Состояние**: Zustand store с 4 слайсами. Все мутации узлов идут через `documentSlice` → `historySlice.setNodes()` (с сохранением undo/redo).
3. **Автосохранение**: После каждого изменения — debounce 1с → запись в Dexie. Есть dirty-save в sessionStorage на случай краша.
4. **Рендер**: `BoardView` рендерит колонки на основе `buildTreeIndex(nodes)`. `TimelineView` — виртуализированный depth-first обход.
5. **Редактирование**: Два режима — Markdown (textarea) и Visual (Tiptap). Переключение без потери данных.
6. **Плагины**: Registry с хуками на создание/обновление/удаление узлов. Пока 1 плагин (voice-fixer).

### Дальнейшее развитие

- **AI-интеграция**: замена mock-провайдера на реальный Gemini API, UI для управления ключами, превью контекста перед отправкой
- **Плагинная платформа**: манифесты, permissions, sandbox — превращение registry в публичную экосистему
- **Производительность**: виртуализация BoardView, diff-based undo для больших документов, lazy-load тяжелых зависимостей
- **CI/CD**: добавление typecheck и format:check в существующий CI
- **Security**: CSP-заголовки, более строгая санитизация HTML, вынос AI-запросов на серверный прокси
- **Accessibility**: полноценный pass по aria, клавиатурной навигации, skip-ссылкам
- **i18n**: 100% покрытие всех видимых строк

## 3. Все найденные проблемы

### 🔴 Критические (P0) — надо исправить

#### P0.1: API-ключи Gemini попадают в клиентский бандл
**Где:** `src/plugins/voice-fixer/utils.ts:54`, `src/components/PluginsPanel.tsx:200`, `.env`
**Кто нашел:** Все 4 аудита
**Проблема:** `VITE_GLOBAL_GEMINI_API_KEY` читается из `import.meta.env` на клиенте и уходит в сборку. При деплое на хостинг ключ оказывается в бандле. Любой может открыть DevTools и использовать его.
**Почему проблема:** Ключ может быть израсходован, превышены квоты, потенциальные расходы.
**Решение:**
- Вариант A (рекомендуемый): Вынести AI-запросы на серверный прокси (Netlify Function, Render сервис и т.д.)
- Вариант B (MVP): В документации явно указать, что публичный ключ — для демо с бесплатным аккаунтом. В интерфейсе требовать собственный ключ.
**Риски решения A:** Нужен сервер, усложнение архитектуры, потеря local-first свойства.
**Риски решения B:** Ключ всё ещё в бандле.
**Можно ли жить:** Сейчас проект локальный. Но при любом публичном деплое — да, это проблема. Статус: **осознанное решение** (см. README).

#### P0.2: localStorage в инициализации store (ломает тесты)
**Где:** `src/store/slices/uiSlice.ts:65`
**Кто нашел:** opencode, MiniMax
**Проблема:** `disabledPlugins: JSON.parse(localStorage.getItem('PUU_DISABLED_PLUGINS') || '[]')` — прямой доступ к localStorage при создании store. В тестовом окружении (jsdom) localStorage может быть недоступен.
**Почему проблема:** Тесты падают или требуют моков. Нарушает принцип: store не должен зависеть от browser API.
**Решение:** Вынести чтение из localStorage в хук `usePreferences.ts` или `useFileSystem.ts`. Store инициализировать с `[]`, а хук на монтировании подтягивает реальное значение.
**Риски:** Низкие. Простой рефакторинг.
**Можно ли жить:** Можно (с моками в тестах), но неправильно архитектурно.

#### P0.3: Нет typecheck и format:check в CI
**Где:** `.github/workflows/ci.yml`
**Кто нашел:** opencode (косвенно)
**Проблема:** CI запускает lint, test, build. Но **не запускает** `npm run typecheck` и `npm run format:check`. Это означает, что ошибки типов и форматирования могут попасть в main.
**Почему проблема:** `typecheck` и `format:check` локально проходят, но нет гарантии, что их не сломают.
**Решение:** Добавить `run: npm run typecheck` и `run: npm run format:check` в steps CI.
**Риски:** Минимальные (оба проходят сейчас).
**Можно ли жить:** Да, если разработчик дисциплинированно запускает их локально.

### 🟡 Высокие (P1) — стоит исправить

#### P1.1: 8 ESLint-ошибок (5 `any` + 1 unused var в voice-fixer + ...)
**Где:** `src/plugins/voice-fixer/utils.ts:6,48,170,142`, `src/components/PluginsPanel.tsx:200`
**Кто нашел:** opencode, GPT OSS
**Проблема:** `@typescript-eslint/no-explicit-any` в коде плагина voice-fixer.
**Почему проблема:** Снижает type-safety, lint падает с ошибками.
**Решение:** Заменить `any` на конкретные типы:
- `xhrFetch` param: `Record<string, unknown>` вместо `any`
- `import.meta as any`: объявить `ImportMeta` интерфейс в `globals.d.ts`
- `catch (e: any)`: убрать `: any`
**Риски:** Низкие.
**Можно ли жить:** Да, но лучше убрать перед публичным деплоем. Статус: **осознанное исключение** для voice-fixer (экспериментальный плагин).

#### P1.2: XHR вместо fetch
**Где:** `src/plugins/voice-fixer/utils.ts:6-28`
**Кто нашел:** Gemini, GPT OSS
**Проблема:** Самодельная обертка `xhrFetch` через `XMLHttpRequest`. Нет нормальной обработки timeout, abort, нет `AbortController`.
**Почему проблема:** Лишний code, хуже читается, нет современных возможностей.
**Решение:** Заменить на `fetch` + `AbortController`.
**Риски:** Низкие. `fetch` — стандарт.
**Можно ли жить:** Да, работает.

#### P1.3: `as Partial<AppStore>` в cross-slice communication
**Где:** `src/store/slices/selectionSlice.ts:17`
**Кто нашел:** opencode, MiniMax
**Проблема:** `return { layoutAlignTrigger: ... } as Partial<AppStore>;` — обход типов Zustand для обновления поля другого слайса.
**Почему проблема:** Хотя это `as Partial<AppStore>`, а не `as any` (т.е. type-safe), это неочевидный паттерн. layoutAlignTrigger — UI-поле, но лежит в слайсе, который меняет selectionSlice.
**Решение:** Перенести `layoutAlignTrigger` в `uiSlice`. Пусть selectionSlice вызывает set из uiSlice.
**Риски:** Средние — может затронуть логику выравнивания.
**Можно ли жить:** Да. Работает стабильно.

#### P1.4: Undo/Redo хранит полные копии массива узлов
**Где:** `src/store/slices/historySlice.ts:89-95`
**Кто нашел:** Все 4 аудита
**Проблема:** Каждый шаг undo = полная копия `PuuNode[]`. `HISTORY_LIMIT = 50`. При 500+ узлах с контентом ~5MB каждый — потенциально ~250MB памяти.
**Почему проблема:** Расход памяти, возможные фризы на больших документах.
**Решение:** Перейти на diff-based / patch-based историю (immer, zundo, или свои патчи). Либо ограничить `HISTORY_LIMIT` для больших документов.
**Риски:** Средние — сложная имплементация, может сломать undo/redo. Требует тщательного тестирования.
**Можно ли жить:** Да, для типовых сценариев (~100-300 узлов). Статус: **известное ограничение**.

#### P1.5: BoardView без виртуализации
**Где:** `src/components/BoardView.tsx`
**Кто нашел:** Все 4 аудита
**Проблема:** Все видимые карточки рендерятся сразу. На >200 карточках — лаги.
**Почему проблема:** Падение производительности на больших документах.
**Решение:** Виртуализация (react-virtuoso или кастомная). Осторожно: не сломать scroll alignment, active path, DnD, floating actions.
**Риски:** Высокие — сложно имплементировать, может сломать много UI-фич.
**Можно ли жить:** Да. Для типовых документов (~50-100 карточек) нормально. Статус: **известное ограничение**.

#### P1.6: Жёстко зашитые model fallback chain
**Где:** `src/plugins/voice-fixer/utils.ts:101-103`
**Кто нашел:** GPT OSS
**Проблема:** `fallbackChain = [preferredModel, 'gemini-2.5-flash', 'gemini-3.1-flash-lite']` — список моделей вшит в код.
**Почему проблема:** Новые модели Gemini не будут использоваться, пока не обновят код.
**Решение:** Вынести в конфиг или получать динамически.
**Риски:** Низкие.
**Можно ли жить:** Да, но периодически нужно обновлять.

**Проблема:** `nodes === _cachedNodes` — проверка по ссылке. Если передать новый массив с теми же данными — cache miss. Если мутировать массив (что редко, но бывает) — cache попадает в stale state.
**Анализ:** `buildTreeIndex` — O(N). Для типовых документов разницы нет. cache даёт выигрыш только если один и тот же объект массива передаётся повторно (что и происходит в реальности — `state.nodes`).
**Решение:** Убрать кэш (простота и корректность) или перейти на content-based hash.
**Риски:** Низкие. Без кэша функция всё ещё O(N).
**Можно ли жить:** Да, кэш работает корректно в текущей архитектуре (всегда передаётся `state.nodes`).

#### P3.4: Сплит карточки не учитывает Markdown-синтаксис

**Решение:** AST-aware split или отключение функции внутри синтаксических конструкций.
**Риски:** Средние — сложность реализации.
**Можно ли жить:** Да, пользователь может поправить вручную.
