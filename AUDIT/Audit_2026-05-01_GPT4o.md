# Аудит проекта PuuNote

**Дата:** 2026-05-01  
**Модель:** GPT-4o (My review)  
**Версия:** 0.4 (Local-first нелинейный редактор)

---

## 1. Описание проекта

PuuNote — это local-first текстовый редактор с горизонтальной ветвящейся (древовидной) структурой карточек. Вместо традиционной вертикальной прокрутки пользователь работает с системой колонок, где каждая колонка представляет уровень вложенности, а карточки — узлы дерева.

### Архитектура и технологический стек

- **Frontend:** React 19 + TypeScript + Vite 6
- **Стилизация:** Tailwind CSS 4 + Motion (Framer Motion)
- **База данных:** Dexie.js (IndexedDB)
- **Маркдаун:** react-markdown + remark-gfm + rehype-sanitize
- **Поиск:** Fuse.js
- **Управление состоянием:** Zustand с subscribeWithSelector middleware
- **Интернационализация:** i18next + react-i18next (RU/EN)

### Ключевые подсистемы

1. **Document Tree API** — изолированные методы манипуляции деревом (documentTree.ts:85-419)
2. **Job Runner** — асинхронные задачи с поддержкой отмены и прогресса (jobRunner.ts)
3. **Plugin Registry** — система хуков и карточных действий (plugins/registry.ts)
4. **AI Foundation** — mock-провайдер и контекстная экстракция для LLM (aiProvider.ts, contextExtraction.ts)
5. **Snapshot System** — макрос undo через снэпшоты в IndexedDB (snapshots.ts)
6. **Timeline View** — линейный режим чтения depth-first обходом

## 2. Удачные решения

### 2.1 Архитектурные

- **Изоляция Document API:** Все операции с деревом проходят через documentApi с валидацией, что предотвращает поломку связей и нарушение React-состояния.
- **Zustand + subscribeWithSelector:** Позволяет подписываться на конкретные части стора без перерисовки всего дерева (`useShallow`).
- **Dexie.js для IndexedDB:** Чистый API поверх IndexedDB с транзакциями и миграциями версий.
- **Snapshot-before-AI:** При AI-операциях создаётся снимок, что защищает от деструктивных мутаций.
- **Debounced autosave:** 1-секундный дебаунс с "dirty save" в localStorage при visibility change — защита от потери данных при закрытии вкладки.

### 2.2 UX-решения

- **Focus Path:** Активный путь подсвечивается, остальные ветки затемняются — минимизация визуального шума.
- **Keyboard-first:** Полноценная навигация стрелками, Tab/Shift+Enter для создания узлов.
- **Smart Clipboard:** Копирование карточки с потомками; multi-select работает с выбранными, не с детьми.
- **Fuzzy Search:** Command Palette с Fuse.js для глобального поиска.

### 2.3 Безопасность

- **XSS-защита:** `rehype-sanitize` + ограничение протоколов для ссылок/изображений.
- **Zod-валидация:** Все импортируемые узлы проходят через PuuNodeSchema.
- **localStorage fallback:** При ошибках записи в IndexedDB — fallback в localStorage ("dirty save").

## 3. Проблемы и рекомендации

### 3.1 Критические (High Priority)

#### Проблема 3.1.1: Race condition при переключении файлов

**Файл:** `useFileSystem.ts:285-334` (`switchFile`)

```typescript
await flushPendingSave();
// ... load new nodes ...
isHydratingFile = true;
useAppStore.getState().setNodesRaw(newNodes);
```

**Проблема:** `isHydratingFile` — глобальная mutable переменная. Если пользователь быстро переключает файлы, могут возникнуть race conditions между load и save. Нет механизма отмены предыдущей операции.

**Рекомендация:** Заменить на `AbortController` или Promise-based queue с отменой предыдущей операции.

---

#### Проблема 3.1.2: Утечка памяти в clipboard cache

**Файл:** `useAppHotkeys.ts:102-130`

```typescript
let lastCopiedCards: {...} | null = null;
const CLIPBOARD_CACHE_MAX_AGE_MS = 2 * 60 * 1000;
```

**Проблема:** Cache живёт 2 минуты, но не очищается при закрытии вкладки. При интенсивном использовании может накопить много данных.

**Рекомендация:** Уменьшить TTL до 30 секунд или добавить LRU-вытеснение с лимитом на количество записей.

---

#### Проблема 3.1.3: Нет валидации при paste в editing mode

**Файл:** `useAppHotkeys.ts:178-266`

**Проблема:** Paste обрабатывается только когда `!editingId`. Но что если пользователь вставляет текст в textarea редактирования? Логика есть, но нет обработки Markdown-импорта внутрь редактируемой карточки.

**Рекомендация:** Добавить интеллектуальный парсинг Markdown при вставке в editing mode — преобразование Markdown-структуры в несколько карточек при вставке в корень (создание children).

### 3.2 Средней важности (Medium Priority)

#### Проблема 3.2.1: Дублирование кода в documentSlice

**Файл:** `documentSlice.ts` — множество мест с паттерном:

```typescript

get().setNodes((prev) => {
  const { nextNodes, newId } = documentApi.XXX(prev, ...);
  newIdValue = newId;
  return nextNodes;
});
```

**Рекомендация:** Вынести в utility function:
```typescript
const applyAndTrackNewId = (operation: (nodes: PuuNode[]) => {nextNodes: PuuNode[], newId?: string}) => ...
```

---

#### Проблема 3.2.2: Неиспользуемый параметр в useBoardLayout

**Файл:** `useBoardLayout.ts:19`

```typescript
unfocusedDepthLimit = Number.POSITIVE_INFINITY,
```

**Проблема:** Параметр есть в сигнатуре, но `useColumns` не использует его — вычисления идут только через `useActiveCorridor`.

**Рекомендация:** Либо реализовать функциональность, либо удалить мёртвый параметр.

---

#### Проблема 3.2.3: "Magic numbers" в UI-логике

**Файл:** `useAppHotkeys.ts:392-397`

```typescript
if (e.key === " ") {
  // Space toggles FloatingCardActions
```

**Проблема:** Space используется для UI-действия, но в редактируемой карточке space может быть частью текста. Есть защита (`if (editingId)`), но UX-решение спорное — space обычно нужен для набора пробела.

**Рекомендация:** Переназначить на другой хоткей (например, `/` или `f`).

---

#### Проблема 3.2.4: Смешанный импорт Markdown

**Файл:** `markdownParser.ts:198-205`

```typescript
const isPuuNoteFormat = mdText.trimStart().startsWith(PUUNOTE_FORMAT_MARKER);
if (isPuuNoteFormat) {
  return parsePuuNoteFormat(mdText);
} else {
  return parseMindMapFormat(mdText);
}
```

**Проблема:** Heuristics для определения формата хрупкие. Markdown с `---` разделителями парсится как PuuNote-формат, хотя это может быть обычный Markdown с горизонтальной чертой.

**Рекомендация:** Добавить более надёжную эвристику или явное указание формата при импорте.

#### Проблема 3.2.5: Тесты покрывают не всё

**Проблема:** Есть тесты для `useBoardLayout.test.ts`, `schema.test.ts`, `documentExport.test.ts`, `documentTree.test.ts`, `contextExtraction.test.ts`, `tree.test.ts`, `markdownParser.test.ts`, но **нет тестов** для:
- documentSlice (операции delete, move, merge)
- useFileSystem (save/load)
- CommandPalette (fuzzy search)
- AI operations
- Snapshot system

**Рекомендация:** Добавить интеграционные тесты для критических пользовательских сценариев.

---

#### Проблема 3.2.6: UI State в глобальном сторе

**Файл:** `uiSlice.ts`

**Проблема:** `theme`, `colWidth`, `inactiveBranchesMode`, `focusModeScope`, `editorMode`, `editorEnterMode`, `pasteSplitMode` — всё в одном слайсе. При изменении любого значения перерисовываются все подписчики.

**Рекомендация:** Разделить на отдельные слайсы (settingsSlice, uiPreferencesSlice) или использовать `useShallow` более точечно.

## 4. UI/UX ошибки и недочёты

### 4.1 Визуальные проблемы

1. **No loading states для async операций:**
   - Импорт файла (`App.tsx:121-153`) — нет skeleton/spinner
   - Command Palette search — нет loading indicator при поиске
   - Timeline load — только fallback text

2. **No empty states для многих компонентов:**
   - Пустой документ — только "+ Add Fragment"
   - Нет empty state для Command Palette results
   - Нет empty state для удаления последней карточки

3. **Accessibility issues:**
   - FloatingCardActions не имеет keyboard navigation
   - Нет ARIA для drag-and-drop (только визуальные индикаторы)
   - Focus trap в CommandPalette работает, но нет skip-link

4. **Inconsistent UX:**
   - Escape в editing mode сохраняет изменения (хорошо), но нет подтверждения при незавершённом действии
   - Undo/Redo не имеют визуальной индикации (количество шагов)

### 4.2 Глюки и edge cases

1. **Drag-drop edge case:** Перетаскивание карточки на саму себя не обрабатывается корректно ( Card.tsx:168 проверяет `sourceId !== node.id`, но не обрабатывает drop на того же родителя).

2. **Split node edge case:** При split пустой карточки создаётся две пустые карточки (documentTree.ts:236-267).

3. **Merge edge case:** При merge удаляются все descendants, даже если merge не имеет смысла семантически.

4. **Scroll sync:** При быстрой навигации scroll может "догонять" с задержкой (useBoardLayout.ts:162-234 — double RAF).

## 6. Видение развития: API для AI-плагинов

### 6.1 Предпосылки

Текущее состояние — это **foundation** (как заявлено в README). Для полноценного AI-плагин API нужно выйти за рамки mock-реализации.

### 6.2 Предлагаемая архитектура

#### 6.2.1 Слои абстракции

```
┌─────────────────────────────────────────┐
│          Plugin Manager (UI)            │
├─────────────────────────────────────────┤
│         Plugin SDK (public API)         │
├─────────────────────────────────────────┤
│    Plugin Runtime (sandbox/execution)   │
├─────────────────────────────────────────┤
│    Provider Interface (AI backends)    │
└─────────────────────────────────────────┘
```

#### 6.2.2 Компоненты

**1. Plugin Manifest Schema:**
```typescript
interface PluginManifest {
  id: string;           // уникальный ID
  name: string;         // человекочитаемое имя
  version: string;      // semver
  description: string;
  author?: string;
  permissions: Permission[];
  entry?: string;       // для внешних плагинов
  actions?: PluginAction[];
  hooks?: PluginHooks;
}
```

**2. Permission Model:**
```typescript
type Permission = 
  | { type: 'read_document' }
  | { type: 'write_document' }
  | { type: 'network', allowedDomains?: string[] }
  | { type: 'storage', maxSize: number }
  | { type: 'ai_provider', providers: string[] };
```

**3. Provider Interface:**
```typescript
interface AiProvider {
  id: string;
  name: string;
  config: ProviderConfig;
  
  // Методы
  complete(prompt: Prompt): Promise<Completion>;
  stream(prompt: Prompt): AsyncIterable<Chunk>;
  embed(text: string): Promise<number[]>;
  
  // Admin
  validateKey(): Promise<boolean>;
  estimateTokens(text: string): number;
  getPricing(): Pricing;
}
```

**4. Plugin Runtime:**
```typescript
class PluginRuntime {
  // Загрузка
  async loadPlugin(manifest: PluginManifest, bundle: PluginBundle): Promise<PluginInstance>;
  
  // Песочница
  runInSandbox<T>(plugin: PluginInstance, fn: (api: PluginAPI) => T): Promise<T>;
  
  // Lifecycle
  enable(pluginId: string): void;
  disable(pluginId: string): void;
}
```

#### 6.2.3 Plugin API (публичный SDK для плагинов):

```typescript
interface PluginAPI {
  // Document access
  getNodes(): PuuNode[];
  updateNode(id: string, content: string): void;
  addChild(parentId: string, content: string): PuuNode;
  
  // Selection
  getActiveNode(): PuuNode | null;
  getSelectedNodes(): PuuNode[];
  
  // AI
  ai: {
    complete(prompt: string): Promise<string>;
    embedSelection(): Promise<number[]>;
  };
  
  // UI
  showNotification(message: string, type: 'info' | 'error'): void;
  registerAction(action: CardAction): void;
  
  // Storage
  getStorage(key: string): unknown;
  setStorage(key: string, value: unknown): void;
}
```

#### 6.2.4 Формат хранения плагинов

- **Встроенные плагины:** JS/TS bundled с приложением
- **Внешние плагины:** Валидированный JSON manifest + inline JS или URL

#### 6.2.5 Безопасность

1. **Sandbox:** Web Workers или iframe sandbox с minimal API
2. **Resource limits:** timeout, memory, network calls
3. **Key management:** encrypted storage, never in localStorage plain
4. **Audit logging:** все plugin actions логируются

### 6.3 Roadmap реализации

| Этап | Задачи | Приоритет |
|------|--------|------------|
| **Phase 1: Provider abstraction** | Вынести mockAiProvider в интерфейс, добавить OpenAI провайдер | High |
| **Phase 2: Plugin manifest** | Создать схему манифеста, валидацию | High |
| **Phase 3: SDK** | DocumentService API, PluginAPI interface | High |
| **Phase 4: Runtime** | Sandboxed execution, lifecycle management | Medium |
| **Phase 5: UI** | Plugin manager panel, enable/disable, settings | Medium |
| **Phase 6: Marketplace** | Публикация, установка, community plugins | Low |

### 6.4 Пример реализации Phase 1 (OpenAI Provider)

```typescript
// src/ai/providers/openai.ts
export class OpenAIProvider implements AiProvider {
  constructor(private apiKey: string, private model: string = 'gpt-4') {}
  
  async complete(prompt: Prompt): Promise<Completion> {
    const response = await fetch('https://api.openai.com/v1/chat/completions', {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${this.apiKey}`,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        model: this.model,
        messages: prompt.messages,
        temperature: prompt.temperature ?? 0.7,
      }),
    });
    
    if (!response.ok) throw new Error(`OpenAI API error: ${response.status}`);
    
    const data = await response.json();
    return { 
      content: data.choices[0].message.content,
      usage: data.usage,
    };
  }
  
  // ... stream, embed, validateKey, getPricing
}
```

## 7. Заключение

PuuNote 0.4 — это **хорошо структурированный прототип** с продуманной архитектурой, local-first подходом и качественным UX foundation. Ключевые сильные стороны:

- Изолированный Document API с валидацией
- Snapshot-based undo система
- Plugin infrastructure (хотя ещё не используется)
