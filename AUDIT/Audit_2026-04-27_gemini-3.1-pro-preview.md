# PuuNote AI - Полный Аудит Проекта (Расширенный)
**Дата:** 27 апреля 2026
**Модель:** Gemini 3.1 Pro Preview (с учетом стороннего аудита)

---

## 1. Описание проекта: Что это и как работает?
**PuuNote** — это локальный горизонтальный древовидный текстовый редактор. Он позволяет организовывать мысли не в виде линейной "простыни", а в виде ветвящихся колонок, что идеально подходит для структурирования информации, outline-заметок и конспектирования.

**Стек технологий:**
- **Frontend Framework**: React 19 + Vite + TypeScript.
- **State Management**: Zustand (глобальный стейт `useAppStore` с реализацией паттерна History для Undo/Redo).
- **Хранилище (DB)**: IndexedDB с использованием библиотеки Dexie (в файле `src/db/db.ts`). Реализована ленивая миграция с предыдущей реализации на `localStorage`.
- **Стайлинг**: Tailwind CSS + `tailwindcss/typography` для рендеринга Markdown-контента.
- **Интерфейс & Взаимодействие**: Command Palette (командная строка для поиска и экшенов), кастомный Drag-n-Drop (через нативный HTML5 API), анимации (Framer Motion), локализация (`react-i18next`). Рендеринг карточек переведен на безопасный компонент `SafeMarkdown`.

**Как устроены данные?**
Дерево узлов (`PuuNode`) хранится в виде "плоского" массива объектов `{ id, parentId, order, content }`.
Это элегантное решение позволяет:
1. Выполнять перемещение карточек простым изменением свойства `parentId`.
2. Избегать ресурсоемких рекурсивных обходов на уровне базы данных.
3. Эффективно осуществлять рендеринг: интерфейс строит колонки слева направо, фильтруя узлы по `parentId` "на лету" с учетом поля `order`.

## 2. Удачные архитектурные решения
1. **Плоская модель данных (Flat State)**: Деревья как нормализованный массив (O(1) / O(n) мутации) — отлично масштабируется. Оптимизированный поиск детей через построение `childrenMap`.
2. **Центрирование фокуса (Active Path Targeting)**: Хорошо сделано авто-скроллирование колонок через расчет `getBoundingClientRect` вслед за курсором.
3. **Разделение режимов View/Edit**: Отображение Read-Only Markdown-превью предотвращает загрязнение экрана полями ввода и защищает текст от случайных правок.
4. **Очистка от старых костылей**: Из кодовой базы успешно вычищены небезопасные одноразовые `repair`-скрипты, устаревший `useHistory` и `EditorContext`. Функции работы с деревом консолидированы в Zustand-сторе.
5. **Безопасность**: Заменен рендеринг на `SafeMarkdown`, убран проброс `GEMINI_API_KEY` в клиентский бандл.

## 3. Найденные проблемы (Уязвимости, ошибки, неоптимальности)

Аудит выявил ряд средних и критически важных проблем во взаимодействии компонентов, производительности и устойчивости данных.

### А. Хранение данных, PWA и Безопасность
1. [x] **Уязвимости в npm: PWA Plugin (High Severity)**
   - **Суть проблемы**: Команда `npm audit` ругается на `serialize-javascript` (цепочка: `vite-plugin-pwa` -> `workbox-build` -> `@rollup/plugin-terser` -> `serialize-javascript`). Присутствует RCE/DoS advisory. Учитывая, что это build-time зависимость, риск ограничен, но для деплоя это ломает пайплайны. К тому же PWA Manifest (`manifest.webmanifest`) ссылается на иконки (`pwa-192x192.png`), которых **нет** в репозитории.
   - **Решение**: Добавить заглушки иконок, либо временно отключить PWA-плагин (дождаться патча). *(PWA временно отключен)*
2. [x] **Потеря изменений при быстром переключении документов (Гонка данных в Dexie)**
   - **Суть проблемы**: Сохранение в `useFileSystem.ts` дебаунсится одним глобальным таймером (на 1000 мс). Если пользователь печатает в файле А и мгновенно переключается на файл В, таймер отменяется, а изменения файла А могут так и не улететь в IndexedDB.
   - **Решение**: При вызове функций `switchFile`, `createNewFile` или удалении делать принудительный (synchronous) `flush` данных старого файла в Dexie, и только потом сменять `activeFileId`.
3. [x] **Ленивая миграция localStorage -> IndexedDB**
   - **Суть проблемы**: Считываются все метаданные файлов, но *содержимое* (`nodes`) мигрируется в Dexie только когда файл становится `active`. Это создает риски частичной потери данных при очистке кэша со стороны пользователя.
   - **Решение**: Миграция вынесена в единый синхронный блок на старте приложения.


### В. UX, Интерфейс и Состояния
5. [x] **Тема оформления не применяется при первичной загрузке**
   - **Суть проблемы**: В `usePreferences.ts` `document.documentElement.classList...` подписывается на изменения стора. Первичный `setState({ theme })` происходит до инициализации подписки, в итоге выбранная в localStorage тема может не примениться к DOM до первого ручного переключения.
   - **Решение**: Вынести логику `applyTheme(theme)` в отдельную утилиту и вызывать её синхронно при чтении из LocalStorage (mounted).
6. [x] **Кнопка Command Palette в Header НЕ работает**
   - **Суть проблемы**: В `Header.tsx` кнопка "Search" делает `document.dispatchEvent(new KeyboardEvent('keydown', { key: 'k', metaKey: true }))`. При этом `CommandPalette` слушает `window.addEventListener`, а у синтетического `KeyboardEvent` `bubbles: false`. Событие просто не долетает до слушателя.
   - **Решение**: Вынести статус модалки `commandPaletteOpen` напрямую в глобальный Zustand-тейт.
7. [x] **Удаление файла через Fake MouseEvent**
   - **Суть проблемы**: `deleteFile` в хуке файловой системы ждет объект `React.MouseEvent`. Command Palette дергает ее, подсовывая "фейковый" объект `{ stopPropagation: () => {} }`. Это классический bad practice.
   - **Решение**: Выделить бизнес-логику `deleteFile(id)` из UI-обертки `handleDeleteClick(e, id)`.
8. [x] **Шумная история Undo/Redo в Zustand**
   - **Суть проблемы**: `AutoSizeTextarea` вызывает `onChange` после дебаунса, а потом дублирует его на `blur`. Даже если текст *не поменялся*, `updateContent` возвращает старые данные, но редьюсер отфильтровывает массив, создавая новый reference `nodes`. Zustand расценивает это как новый шаг истории (Undo). При больших ветках Undo-стейт быстро пожирает мегабайты оперативной памяти.
   - **Решение**: Прерывать `setNodes`, если глубокого изменения не было (`if (isEqual(old, new)) return;`).
9. [x] **Виртуализация и DOM-утечки в App.tsx (`colRefs.current`)**
   - **Суть проблемы**: При сжатии дерева массив `colRefs.current` не обрезается, сохраняя ссылки на умершие DOM узлы. Более того, навигация жестко завязана на структуру DOM (`col.children[0].children[0]`), что сломается при малейшем изменении верстки.
   - **Решение**: Добавлена очистка ссылок `colRefs.current.length = columns.length` и безопасный `querySelector` вместо `children`.

### C. Разбор Markdown и Сложности форматов
10. [x] **Экспорт маркер в Markdown дублируется (`<!-- puunote-format: 1 -->`)**
    - **Суть проблемы**: `exportNodesToMarkdown` добавляет спец. комментарий вверх файла. При импорте парсер использует split, но этот комментарий не вырезается и застревает случайным текстом в качестве содержимого первой карточки. И так по кругу при каждом импорте/экспорте.

## 4. Дальнейшее развитие и План действий (Next Steps)

Текущая кодовая база значительно чище, чем раньше, однако она перекошена в сторону расширения фич, в то время как "каркас" (сохранение/история/события) страдает от мелких инженерных недочетов.

**Архитектурные Рефакторинги (Этап 2)**
7. **Переход на настоящие CRDT-структуры (Yjs / Automerge)**: Это автоматически решит проблему разрастания `past/future` в Undo, а также подготовит фундамент для будущего коллаборативного редактирования.
8. **Замена HTML5 Drag&Drop**: Использовать библиотеку вроде `@dnd-kit/core`. Текущий draggable слишком "хрупок".
9. **Виртуализация отображения (Virtuoso)**: При рендере 500+ карточек DOM уходит в фризы. Необходима реализация Windowing-а (виртуальный рендеринг списка в колонках). 
10. [x] **Prettier & Линтеры**: Включить `format:check` в CI – прямо сейчас `npx prettier --check src/` выдает ошибки на 6 файлах. Убрать `allowJs: true` из `tsconfig.json`.
