## 1. Исполнительное резюме

PuuNote сейчас выглядит как сильный local-first прототип горизонтального древовидного Markdown-редактора. Основа хорошая: плоская модель дерева, доменный API для операций над карточками, Zustand-состояние, IndexedDB через Dexie, импорт/экспорт Markdown/JSON, снапшоты и начальный AI/job/plugin слой.

Главная проблема не в идее и не в общей архитектуре, а в инженерной надежности текущего состояния:

- `npm run build` сейчас падает на TypeScript-ошибке в `AutoSizeTextarea`.
- `npm run test` сейчас падает на конфликтующем тесте `useBoardLayout`.
- Есть реальные edge cases сохранения: пустой документ не сохраняется и не загружается как пустой, последние символы из textarea могут не попасть в store перед закрытием вкладки, а `visibilitychange:hidden` отменяет pending IndexedDB-save.
- Часть UX-контрактов не совпадает с интерфейсом: горячие клавиши, tutorial detection, `editorMode`, поведение Timeline.
- Мобильный header статически перегружен набором кнопок; это подтверждает вывод аудитов о риске недоступной кнопки Settings на узком экране.
- Линтер чистый и `npm audit` не нашел известных уязвимостей, но форматирование не проходит `prettier --check`, зависимостям нужны отдельные контролируемые обновления.

Коротко: сначала нужно стабилизировать build/test/persistence, затем полировать UX/i18n/accessibility, и только после этого расширять AI/plugin возможности.

## 2. Что это сейчас за проект

PuuNote - браузерный local-first редактор заметок и текстовых структур. В отличие от обычного вертикального документа, пользователь работает с деревом карточек: главные мысли слева, уточнения и дочерние ветки раскрываются вправо в соседних колонках. По продуктовой метафоре проект ближе к Gingko App, mind-map редактору и Markdown outline editor, но с акцентом на текстовые карточки.

Типичные сценарии:

- планирование статей, книг, сценариев, проектов;
- конспекты с вложенными пояснениями;
- разбор сложных тем с ветвлениями;
- нелинейное редактирование текста;
- импорт/экспорт Markdown и JSON-бэкапы;
- будущие AI-операции над выделенной веткой.

## 3. Как проект устроен

### Технологии

- React 19, TypeScript, Vite 6.
- Tailwind CSS 4 и `@tailwindcss/typography`.
- Zustand для глобального состояния.
- Dexie/IndexedDB для local-first хранения документов, карточек и снапшотов.
- `react-markdown`, `remark-gfm`, `remark-breaks`, `rehype-sanitize` для Markdown.
- `react-virtuoso` для виртуализированного Timeline view.
- Fuse.js для поиска.
- Zod для валидации импортируемых и сохраненных данных.
- Sonner для toast-уведомлений.

### Данные

Основная модель - плоский массив `PuuNode[]`.

Каждая карточка содержит:

- `id`;
- `content`;
- `parentId`;
- `order`;
- необязательные `metadata`.

Документ хранится отдельно как `PuuDocument`:

- `id`;
- `title`;
- `updatedAt`;
- необязательные `metadata`.

Дерево восстанавливается через `parentId` и `order`. Для быстрых операций строится `TreeIndex`: `nodeMap` и `childrenMap`.

### Основные слои

- `src/store/useAppStore.ts` собирает Zustand store из слайсов.
- `src/store/slices/documentSlice.ts` содержит команды над документом: add/delete/move/split/merge/export.
- `src/store/slices/historySlice.ts` хранит undo/redo историю и эмитит plugin hooks.
- `src/store/slices/selectionSlice.ts` управляет active/selected/editing/drag/fullscreen состояниями.
- `src/store/slices/uiSlice.ts` хранит настройки UI.
- `src/domain/documentTree.ts` - доменный API дерева карточек.
- `src/domain/documentService.ts` - IndexedDB, миграция legacy localStorage, нормализация, поиск.
- `src/hooks/useFileSystem.ts` - инициализация, переключение файлов, автосохранение, dirty backup.
- `src/utils/markdownParser.ts` - Markdown import/export и clipboard round-trip.
- `src/components/BoardView.tsx` - основное многоколоночное дерево.
- `src/components/Card.tsx` - карточка, редактирование, drag & drop.
- `src/components/TimelineView.tsx` - линейный виртуализированный вид.
- `src/components/FileMenu.tsx`, `CommandPalette.tsx`, `SettingsPanel.tsx`, `SnapshotPanel.tsx` - основные панели.
- `src/domain/aiProvider.ts`, `aiOperations.ts`, `jobRunner.ts`, `plugins/registry.ts` - экспериментальный фундамент AI/plugin сценариев.

### Как это работает в runtime

1. При старте `useFileSystemInit` мигрирует legacy localStorage, читает список документов из IndexedDB и загружает активный документ.
2. `BoardView` строит `TreeIndex`, active path и колонки через `useBoardLayout`.
3. Клик по карточке делает ее активной, double click или Enter включает редактирование.
4. `AutoSizeTextarea` держит локальный текст и с debounce 150 ms отправляет изменения в Zustand.
5. Zustand `setNodes` кладет предыдущий массив в undo history и эмитит plugin events.
6. `useFileSystem` подписан на изменения `nodes` и с debounce 1000 ms сохраняет карточки в IndexedDB.
7. При `beforeunload`, `pagehide`, `visibilitychange:hidden` создается dirty backup в `sessionStorage`.
8. Markdown/JSON export скачивает файл через браузерный `download`.
9. Clipboard использует plain Markdown, HTML с meta payload и попытку кастомного MIME.
10. AI-команда сейчас использует mock provider, делает snapshot и добавляет дочерние draft-карточки.

## 6. Подтвержденные проблемы

### 6.1. Сборка, тесты, релизная готовность

#### P0-1. Проект сейчас не собирается

- Где: `src/components/AutoSizeTextarea.tsx:115`.
- Актуальность: подтверждено командой `npm run build`.
- Что происходит: `react-textarea-autosize` ожидает тип `Style`, где `height` должен быть `number`, а компонент передает объект как `React.CSSProperties`, где `height` может быть строкой.
- Почему это проблема: production build и Render build не пройдут. Любой деплой через `render.yaml` с `buildCommand: npm ci && npm run build` будет остановлен.
- Возможное решение: типизировать `style` через типы `ReactTextareaAutosize`, убрать конфликтующую возможность строкового `height`, либо аккуратно привести только допустимые style-поля.
- Риски реализации: можно сломать внешний `style` API компонента, если где-то реально передается строковая высота; после фикса нужно проверить карточку, Timeline и fullscreen editor.
- Критичность: P0.
- Можно не трогать сейчас: нет.

#### P0-2. Тестовый suite не зеленый

- Где: `src/hooks/useBoardLayout.test.ts:48-61`.

- Возможное решение: выбрать продуктовый контракт. По текущему help/UI логичнее, что cleared focus показывает все дерево, значит root-only тест нужно удалить или переписать под явный отдельный режим.
- Риски реализации: если root-only поведение кому-то нужно, его надо делать отдельным параметром, иначе можно снова получить скрытую UX-регрессию.

### 6.2. Сохранение и целостность данных

#### P1-1. Пустой документ не сохраняется и не загружается как пустой

- Где: `src/hooks/useFileSystem.ts:91-110`, `196-207`, `266-270`, `317-325`; `src/domain/documentService.ts:185-208`.
- Актуальность: подтверждено по коду.
- Что происходит: hooks выходят раньше при `nodes.length === 0`. Даже если пустой массив когда-то попадет в IndexedDB, загрузка трактует `[]` как отсутствие данных и подставляет `INITIAL_NODES` или считает файл ошибочным.
- Почему это проблема: пользователь может удалить последнюю карточку, увидеть пустой документ, но после reload получить старое содержимое или tutorial/default nodes. Это прямой риск потери доверия к сохранению.
- Возможное решение: различать "файл не найден" (`null`) и "файл найден, но пустой" (`[]`); разрешить saveNodes(fileId, []); убрать `nodes.length === 0` guards из автосохранения и dirty backup.
- Риски реализации: UI должен корректно работать с пустым `nodes`; нужно проверить BoardView, TimelineView, snapshots, export, activeId fallback.
- Критичность: P1.
- Можно не трогать сейчас: нежелательно. Это не каждый день, но это data correctness.

#### P1-2. Последние символы из textarea могут не попасть в store перед закрытием вкладки

- Где: `src/components/AutoSizeTextarea.tsx:42-87`; `src/hooks/useFileSystem.ts:266-290`.
- Актуальность: подтверждено по коду.
- Что происходит: textarea держит локальный `localValue` и отправляет изменения в Zustand через debounce 150 ms. `beforeunload/pagehide` читают `useAppStore.getState().nodes`, где последние символы еще могут отсутствовать.
- Почему это проблема: пользователь может напечатать текст и сразу закрыть/перезагрузить вкладку. Dirty backup сохранит старую версию store, а не актуальный DOM textarea.
- Возможное решение: сделать централизованный flush pending editor changes перед dirty backup; либо обновлять Zustand на каждый input, а debounce оставить только для IndexedDB-save.
- Риски реализации: обновление store на каждый input увеличит нагрузку и размер undo history; лучше одновременно пересмотреть batching истории.
- Критичность: P1.
- Можно не трогать сейчас: лучше не откладывать, потому что это риск потери последних правок.

#### P1-3. `visibilitychange:hidden` отменяет pending IndexedDB-save

- Где: `src/hooks/useFileSystem.ts:266-284`.
- Актуальность: подтверждено по коду.
- Что происходит: при скрытии вкладки вызывается `saveCurrentStateToDirtyBackup`, который делает `fsManager.clearTimer()`. Pending save в IndexedDB отменяется, а данные остаются только в `sessionStorage`.
- Почему это проблема: если вкладка была просто свернута, полноценное сохранение в IndexedDB не произойдет. При некоторых сценариях закрытия/сбоя `sessionStorage` - слабая замена.
- Возможное решение: на hidden вызывать `flushPendingSave()` или сохранять в IndexedDB немедленно; dirty backup оставить как дополнительную страховку.
- Риски реализации: async-save в `visibilitychange` может не успеть; нужен fallback через dirty backup, но без отмены надежного save без причины.
- Критичность: P1.
- Можно не трогать сейчас: лучше не откладывать.

#### P1-4. `updatedAt` не обновляется при обычном редактировании текста

- Где: `src/hooks/useFileSystem.ts:234-250`; сортировка в `src/components/FileMenu.tsx:64-66`.
- Актуальность: подтверждено по коду.
- Что происходит: `updatedAt` обновляется только если изменился title, выведенный из первой карточки. Редактирование тела документа без изменения title не меняет timestamp.
- Почему это проблема: список документов сортируется по устаревшему времени, пользователь не видит реальный порядок последних изменений.
- Возможное решение: обновлять `documents[].updatedAt` после успешного `DocumentService.saveNodes`, независимо от изменения title.
- Риски реализации: если обновлять metadata слишком часто, будет лишняя запись `documents`; стоит делать в том же debounce, что и nodes-save.
- Критичность: P1/P2.
- Можно не трогать сейчас: можно ненадолго отложить, но это заметная UX/data metadata ошибка.

#### P2-2. Dirty backup и snapshots не поддерживают пустой документ

- Где: `src/hooks/useFileSystem.ts:269-270`; `src/db/snapshots.ts:80-83`.
- Актуальность: подтверждено по коду.
- Что происходит: dirty backup пропускает `[]`, restore snapshot считает пустой snapshot invalid.
- Почему это проблема: если пустой документ становится валидным состоянием, эти слои будут конфликтовать с новым контрактом.
- Возможное решение: после решения P1-1 обновить dirty backup и snapshots под пустой документ.
- Риски реализации: нужно явно решить продуктовый контракт: пустой документ допустим или система всегда создает хотя бы одну пустую карточку.
- Критичность: P2, но связано с P1-1.
- Можно не трогать сейчас: да, если сначала не меняется контракт пустых документов.

### 6.3. Tutorial, документы и импорт/экспорт

#### P1-5. Tutorial detection завязан на неправильный title

- Критичность: P1.
- Можно не трогать сейчас: можно, если кнопкой tutorial почти не пользуются; но это простой и заметный UX баг.

#### P2-3. Title документа выводится из первой карточки без явного ручного переименования

- Риски реализации: нужно мигрировать существующие документы и аккуратно совместить auto-title с manual-title.
- Критичность: P2.

- Почему это проблема: derived index выглядит как immutable структура, но может меняться побочным эффектом. Это создает хрупкость при reuse cache.

- Риски реализации: если убрать cache без замены, можно ухудшить performance при частых вызовах.
- Критичность: P2.
- Можно не трогать сейчас: можно, но fix сортировки копии дешевый.

## 7. Частично подтвержденные или снятые пункты из исходных аудитов

## 10. Общая оценка

PuuNote уже имеет хороший каркас продукта: local-first хранение, понятная древовидная модель, Markdown sanitation, импорт/экспорт, снапшоты и начальный AI/job слой. Но текущее состояние нельзя считать стабильным для релиза: build и test красные, а persistence содержит edge cases, которые для редактора заметок критичнее большинства UI-недочетов.

Лучший путь развития - не добавлять новые крупные функции прямо сейчас, а сначала сделать проект надежным: зеленые проверки, предсказуемое сохранение, мобильная доступность базовых действий и CI. После этого можно спокойно наращивать UX, большие документы и AI/plugin слой.
