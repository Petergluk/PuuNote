# Аудит кодовой базы PuuNote (Версия 2)

**Дата:** 2026-04-27
**Анализатор:** Gemini 3.1 Pro Preview

## 1. Описание проекта
**PuuNote** — это приложение для заметок, ориентированное на иерархическую/column-based архитектуру карточек. Данные (фрагменты текста в формате Markdown) разбиты на узлы (nodes), связанные отношениями parent-child. 
Приложение использует:
- **React + Vite** в качестве основы фронтенда.
- **Zustand** для глобального управления состоянием (включая Undo/Redo и хранение графа узлов).
- **Dexie (IndexedDB)** для постоянного хранения данных (off-line first).
- **Tailwind CSS + motion/react** для стилизации и анимаций.
- **Lucide React** для иконок.

Интерфейс представляет собой горизонтально скроллируемые колонки (а-ля Miller Columns), где каждая колонка — это следующий уровень вложенности графа, а также линейный Timeline View.

## 2. Удачные решения
- **Миграция на IndexedDB**: Переход с localStorage на IndexedDB посредством Dexie снимает критический лимит в 5MB. Текущая асинхронная миграция аккуратно подхватывает старые данные.
- **Блокировка лишних обновлений Undo/Redo**: Логика `hasChanged` в `updateContent` внутри Zustand отсекает создание фиктивных точек истории при блюре или неизмененном вводе.
- **Делинк бизнес-логики и UI**: Хендлеры файловой системы (`useFileSystemActions`) вынесены из компонентов, что упростило переиспользование (например, через Command Palette).
- **Очистка памяти (Memory management)**: Сброс мертвых ссылок `colRefs.current` и безопасный `querySelector` убрали потенциальный краш при навигации из-за висящих узлов DOM.

## 3. Найденные проблемы и неоптимальности

### А. Производительность (Performance & DOM)
1. **Отсутствие виртуализации списков (Критично для масштабирования)**
   - **Симптом**: При рендере документа с 500+ узлами браузер начнет тормозить, так как в `App.tsx` и `TimelineView.tsx` каждый `PuuNode` рендерится как отдельный DOM-узел вместе со своим Markdown-парсером. Заявленная в прошлом аудите галочка `[x]` закрыла лишь утечку памяти, но не саму виртуализацию.
   - **Решение**: Интегрировать библиотеку виртуального рендеринга для списков в `TimelineView` и внутри каждой колонки в `App.tsx`.
2. **Множественные `requestAnimationFrame` в App.tsx**
   - **Симптом**: Синхронизация скролла использует цепочки двойных `requestAnimationFrame`, что выглядит как костыль для обхода таймингов рендеринга React. 
   - **Решение**: Заменить на `useLayoutEffect` или нормально настроенный `useEffect`, либо делегировать логику скролла внутрь самих компонентов колонок.
3. **Мутация Refs в фазе Render**
   - **Симптом**: Конструкция `colRefs.current.length = columns.length;` прямо в теле компонента `App` — это антипаттерн React (особенно для StrictMode и Concurrent Mode). 
   - **Решение**: Управлять чисткой массива в `useLayoutEffect` или управлять им строго через callback (убирая мертвые узлы при демонтировании).

### B. Форматы данных и Хранение
4. **Сохранение графа "целиком" при каждом вводе**
   - **Симптом**: Из-за особенностей Zustand весь массив `nodes` (даже если он весит 1+ MB) пересчитывается и отправляется на сохранение в IndexedDB с дебаунсом в 1 секунду. При огромных файлах это будет медленно изнашивать локальную базу и кратковременно загружать CPU.
   - **Решение**: Нормализовать данные (`{ [id: string]: PuuNode }`) вместо одномерного массива или сохранять дельты изменений (CRDT-подход с Yjs/Automerge).

### C. UX и Архитектура кода
5. **I18n (Локализация)**
   - **Симптом**: В `CommandPalette` используется `useTranslation`, но в `TimelineView.tsx` строки жестко захардкожены на русском (`"Скопировано!"`, `"Скопировать всё"`, `Outline`). Возникает мультиязычный конфликт.
   - **Решение**: Унифицировать систему переводов. Вынести все локализованные строки.
6. **HTML5 Drag and Drop**
   - **Симптом**: Опора на нативные события `onDragStart`/`onDrop` ограничивает возможности перетаскивания (особенно на мобильных устройствах — touch events не работают с нативным HTML D&D без полифиллов).
   - **Решение**: Переход на мобильно-совместимую библиотеку вроде `@dnd-kit/core`.

## 4. Дальнейшее развитие проекта

1. **Рефакторинг State Management (Нормализация данных)**
   - Сейчас `nodes` — это плоский массив. Поиск родителя/ребенка и построение пути происходит через обход массива `for/filter` (O(N) операции в рендере). Нужно переделать стейт в словарь (Map/Record), что сделает обходы дерева O(1).
2. **Mobile UX & Возврат PWA**
   - Вернуть PWA (обновив зависимости, добавив 192x192 иконки), чтобы приложение могло кэшироваться и работать полностью оффлайн на телефоне как нативное.
   - Доработать адаптивность: на 100% избавиться от hover-стейтов, завязанных на логику мыши; докрутить touch-дрэгпикинг.
3. **Rich Media Support (Картинки)**
   - Прикрутить возможность вставлять картинки (буфер обмена / Drag&Drop) с сохранением их напрямую в IndexedDB как Blob, чтобы отказаться от внешних хостингов изображений.
4. **Коллаборативный режим**
   - Интеграция с WebSocket + Yjs. Приложение по своей природе идеально ложится под риал-тайм совместное редактирование (каждый пользователь может быть в своей ветке дерева).
