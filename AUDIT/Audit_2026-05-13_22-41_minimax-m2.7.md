## 1. Project Overview

### 1.1 What is PuuNote?

**PuuNote** — это локально-first горизонтальное tree-based приложение для редактирования Markdown-заметок. Приложение построено как single-page SPA на React 19 + TypeScript + Vite, не требует сервера для работы, данные хранятся в IndexedDB (через библиотеку Dexie) в браузере пользователя.

### 1.2 Core Concept

Документ представляет собой **дерево карточек (nodes)**. Корневые карточки располагаются слева, дочерние — расширяются вправо в виде колонок. Это позволяет визуально организовыровать идеи нелинейно — планы, скрипты, исследования, структуры.

Поддерживается два режима редактирования: классический Markdown (textarea) и визуальный (Tiptap/WYSIWYG). Данные автоматически сохраняются в IndexedDB с debounce в 1 секунду.

### CRITICAL (must fix)

#### C-1: Global mutable cache in `tree.ts` causes data corruption

**File:** `src/utils/tree.ts:8-29`

```typescript
let _cachedNodes: PuuNode[] | null = null;
let _cachedTreeIndex: TreeIndex | null = null;

export const buildTreeIndex = (nodes: PuuNode[]): TreeIndex => {
  if (nodes === _cachedNodes && _cachedTreeIndex) {
    return _cachedTreeIndex;
  }
  // ... builds and caches
  _cachedNodes = nodes;
  _cachedTreeIndex = result;
  return result;
};
```

**Problem:** `buildTreeIndex` uses a **module-level mutable cache** that stores a reference to the input array. This cache is never invalidated when nodes are mutated, but in this codebase nodes are always replaced (never mutated in place), so the cache reference comparison (`nodes === _cachedNodes`) can incorrectly match stale data.

**Worse:** Because `_cachedNodes` holds a reference to the array, it **prevents GC** of old node arrays. For large documents, this creates a memory leak.

**Also:** The cache is only valid for the **single most recently computed index**. If two components call `buildTreeIndex` with different node arrays (e.g., before and after a state update during render), both get the same cached result — which may be wrong for the second caller.

**Impact:**
- Silent data corruption when two different tree computations share the cache
- Memory leak for large documents (5MB+ node arrays are kept in memory)
- Unpredictable behavior when `nodes` array reference happens to match cached reference

**Fix:** Remove the cache entirely, or make it per-call-context (e.g., WeakMap keyed by nodes array identity). Given that tree rebuilding is O(n) and nodes are typically <1000, caching is unnecessary complexity here.

#### C-2: `useColumns` dependency array missing `treeIndex`

**File:** `src/hooks/useBoardLayout.ts:74-86`

```typescript
export function useColumns(
  nodes: PuuNode[],
  treeIndex?: TreeIndex,
  activePath: string[] = [],
  activeId: string | null = null,
  useActiveCorridor = false,
) {
  const columns = useMemo(() => {
    const index = treeIndex || buildTreeIndex(nodes);
    return buildBoardColumns(index, activePath, activeId, useActiveCorridor);
  }, [activeId, activePath, nodes, treeIndex, useActiveCorridor]);
  //    ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
  //    treeIndex can be undefined, but it's in deps!
  return columns;
}
```

**Problem:** `treeIndex` is in the dependency array but is **optional and can be `undefined`**. When `treeIndex` is undefined, `useMemo` recomputes (good). But when the caller passes a `treeIndex`, the memo correctly tracks it. The actual bug is subtler: when `treeIndex` is passed as `undefined` (from callers that don't have it), `useMemo` recomputes every render because `undefined !== undefined` is fine... but the real issue is the `||` fallback — if the caller passes a stable `treeIndex` but `nodes` changes, the cache rebuild correctly handles both cases.

Wait — actually looking more carefully: when `treeIndex` is passed and stable, and `nodes` changes, `buildTreeIndex(nodes)` is called which has its own global cache. But if `nodes === _cachedNodes`, it returns the stale cached index from a PREVIOUS `treeIndex` computation. This is a cascade of C-1.

**Impact:** Stale column layout when `nodes` reference changes but content is similar (e.g., after undo/redo). The board columns may not update correctly.

**Fix:** Fix C-1 first. Then verify `useColumns` works correctly.

#### C-3: `renderInlineMarkdownToHtml` uses string replacement instead of proper AST parsing — XSS risk

**File:** `src/utils/markdownParser.ts:162-182`

```typescript
const renderInlineMarkdownToHtml = (value: string) => {
  const codeSpans: string[] = [];
  let html = escapeHtml(value).replace(/`([^`]+)`/g, (_match, code) => {
    const token = `@@PUUNOTE_CODE_${codeSpans.length}@@`;
    codeSpans.push(`<code>${code}</code>`);
    return token;
  });

  html = html
    .replace(/\*\*([^*]+)\*\*/g, "<strong>$1</strong>")
    // ... more regex replacements
    .replace(/\[([^\]]+)\]\((https?:\/\/[^)\s]+)\)/g, '<a href="$2">$1</a>');
```

**Problem:** This custom inline Markdown-to-HTML renderer is used for **clipboard HTML export** (`markdownToClipboardHtml`). It applies `escapeHtml` first, BUT the link replacement `replace(/\[([^\]]+)\]\((https?:\/\/[^)\s]+)\)/g, ...)` puts raw URL into `href` without sanitization. While `escapeHtml` prevents XSS in the link text, a malicious URL like `javascript:alert(1)` would NOT be caught by this regex (it requires `http`/`https`). However, this is only for clipboard HTML export, not for rendering.

The bigger concern: the regex replacements are order-dependent and can be broken by edge cases (nested formatting, escaped characters). This is fragile and doesn't match the complexity of real Markdown.

**But there's a critical issue:** In `markdownToClipboardHtml` (line 292-297), this is used to generate clipboard HTML. If a user copies nodes and pastes into a vulnerable context, the `javascript:` URL bypass is mitigated because the regex only matches `http`/`https`.

However — the function is also exported as `exportNodesToClipboardHtml`. The clipboard HTML includes both a meta tag with JSON data AND the rendered HTML. The JSON path (`parseClipboardNodes`) validates nodes, so that's safe. But the HTML path uses the custom renderer.

**Is this a real XSS?** The regex `https?:\/\/[^)\s]+` is restrictive. But `escapeHtml` is applied before regex replacements, so characters like `<`, `>`, `"` are escaped. The URL is put into the href as-is, not escaped. If a user puts a URL with a newline, the regex might misbehave but not create XSS because of the `\s` constraint.

**Verdict:** This is a **medium-risk potential XSS** in the clipboard HTML export path. The URL sanitization only allows `http`/`https`, and HTML is escaped first, so direct XSS is blocked. However, the approach is fragile and could be bypassed with creative URLs.

**Fix:** Use the same `rehype-sanitize` pipeline that `SafeMarkdown` uses, or at minimum validate URLs strictly.

#### C-4: `FileSystemManager` is a mutable singleton class — race conditions in multi-tab

**File:** `src/hooks/useFileSystem.ts:11-46`

```typescript
class FileSystemManager {
  public fileId: string = "";
  public nodes: PuuNode[] = [];
  public isHydratingFile = false;
  public switchController: AbortController | null = null;

  public scheduleSave(fileId, nodes, onSave) {
    if (this.timer) clearTimeout(this.timer);
    this.fileId = fileId;
    this.nodes = nodes;  // mutable shared state!
    const scheduledFileId = fileId;
    const scheduledNodes = nodes;
    this.timer = setTimeout(() => {
      this.timer = null;
      onSave(scheduledFileId, scheduledNodes);
    }, 1000);
  }
}

export const fsManager = new FileSystemManager();
```

**Problem:**
1. `fsManager` is a **module-level singleton**. If the app is opened in multiple tabs, each tab shares the same `fsManager` instance in memory (same JS context per tab — actually no, each tab is independent. So this is fine per-tab).
2. **But:** The `isHydratingFile` flag is mutable state on the class. If `useFileSystemInit` is called multiple times (e.g., React StrictMode double-invocation), or if the component remounts, the state persists incorrectly.
3. The `nodes` field is a direct reference to the store's nodes array. If the store mutates (it shouldn't, per design), `fsManager.nodes` would be corrupted.

**The real issue:** `isHydratingFile = true` is set, then `setNodesRaw` is called, then `isHydratingFile = false` in a finally block. But the subscription callback at line 226-270 also reads `state.nodes` and `state.activeFileId`. Between `setNodesRaw` and the subscription firing, `isHydratingFile` might already be `false` if the subscription fires during the sync execution of `setState`. The check at line 238 `if (fsManager.isHydratingFile) return` happens AFTER state has already been set.

This means **during hydration, the subscription callback might still trigger a save**, overwriting the hydration data.

**Impact:**
- Data loss: hydration loads old document, subscription immediately saves it (potentially with wrong data)
- Race condition between hydration and auto-save subscription

**Fix:** Add a more robust hydration guard. Consider using a version/counter instead of a boolean, or moving the guard to check against a stored "hydration timestamp."

#### H-1: `historySlice.ts` line 59 — identity comparison is wrong

**File:** `src/store/slices/historySlice.ts:58-66`

```typescript
if (
  currentNodes.length === nextNodes.length &&
  currentNodes.every((node, index) => node === nextNodes[index])
) {
  return; // Returns UNDEFINED!
}
```

**Problem:** When this condition is true, `setNodes` returns `undefined` (no return value). But the Zustand store's `set` function expects the callback to return the partial state to merge. Returning `undefined` from the updater function causes **Zustand to not update the store at all** — the state remains unchanged.

But wait — this is intentional: if nodes haven't changed, don't add to history and don't re-render. However, `return` without a value returns `undefined`, and Zustand's callback convention is that `undefined` means "no state change". This works correctly here because returning undefined from the updater IS a valid signal to Zustand to skip the update.

BUT — the issue is that `setNodes` as defined in `AppStore` is typed as returning `void`. The actual function returns `undefined` intentionally to signal "no change". This is a valid Zustand pattern but it's confusing and error-prone.

**The real bug:** This identity check (`node === nextNodes[index]`) compares by reference. For tree operations that always create new arrays/objects, this should work. But if the SAME array reference is passed (from somewhere that didn't create a new array), the function returns `undefined` and the store does nothing. This can mask bugs where the caller passed the same reference.

More importantly: after the check, if `shouldGroup` is true, the function DOES return `undefined` in the `set` callback — meaning no state change is dispatched even though `nodes` reference changed. This means the store's `nodes` field is NOT updated when history grouping is active!

```typescript
set({
  nodes: nextNodes,  // <-- this IS set when shouldGroup is false
  past: shouldGroup ? state.past : [...state.past.slice(-(HISTORY_LIMIT - 1)), currentNodes],
  future: [],
});
```

Wait, no — `set({...})` is called regardless. The `past` is only preserved when `shouldGroup` is true. So the state IS updated with `nextNodes`. The identity check is an early exit.

**The actual issue:** When `shouldGroup` is true, `state.past` is NOT updated (uses the same array), and `future` is cleared. The nodes ARE updated. This seems fine.

**Re-evaluating:** The bigger issue is that `setNodes` returning `undefined` is a confusing pattern, and the identity comparison at line 63 is unreliable. If `nextNodes` is a new array but contains the same node references, this returns early — skipping history capture but also skipping the actual Zustand state update... Wait, no — look at line 84-90:

```typescript
set({
  nodes: nextNodes,
  past: shouldGroup ? state.past : [...state.past.slice(-(HISTORY_LIMIT - 1)), currentNodes],
  future: [],
});
```

This is inside the `set({...})` call. So state IS updated. The `return` at line 66 is inside the `if` check BEFORE the `set`. It prevents the `set` from being called at all when nodes haven't changed.

So the early return correctly prevents redundant history entries when nothing changed. The concern is only if the identity check incorrectly matches (same node references but logically different content) — but that can't happen because node content changes always create new node objects.

**Verdict:** Actually not a bug. But the pattern is fragile.

#### H-5: Document title derivation from content can throw on malformed input

**File:** `src/hooks/useFileSystem.ts:50-80`

```typescript
const cleanTitle = (value: string) => {
  const stripped = value
    .replace(/^#{1,6}\s+/, "")
    .replace(/^[-*+]\s+/, "")
    .replace(/^>\s*/, "")
    .replace(/^[*_`~]+|[*_`~]+$/g, "")
    .trim();
  const bracketMatch = stripped.match(/^\[(.+)\]$/);
  return (bracketMatch?.[1] || stripped).trim();
};

export const deriveDocumentTitle = (nodes, fallback = "Untitled") => {
  const roots = nodes
    .filter((node) => node.parentId === null)
    .sort((a, b) => (a.order || 0) - (b.order || 0));
  const firstNode = roots[0] || nodes[0];
  if (!firstNode) return fallback;

  const firstMeaningfulLine =
    firstNode.content
      .split("\n")
      .map((line) => line.trim())
      .find((line) => line.length > 0) || "";
  const title = cleanTitle(firstMeaningfulLine);
  if (!title) return fallback;

  return title.length > 80 ? `${title.slice(0, 77)}...` : title;
};
```

**Problem:** If `firstNode.content` is extremely large (5MB limit), calling `.split("\n")` creates millions of lines, which is extremely slow and can freeze the browser.

**Impact:** Performance freeze when a node has massive content and the document title needs to be derived (e.g., on save).

**Fix:** Limit the content scanned for title extraction. Only scan the first 1000 characters.

#### H-6: `normalizeNodesWithReport` — duplicate ID regeneration is O(n²)

**File:** `src/domain/documentService.ts:94-128`

```typescript
for (const node of validatedNodes) {
  let id = node.id;
  while (seenIds.has(id)) {  // O(n) per node in worst case
    id = generateId();       // generateId uses crypto.randomUUID — slow
    report.repaired = true;
    report.warnings.push(`Regenerated duplicate node id "${node.id}".`);
  }
  seenIds.add(id);
  // ...
}
```

**Problem:** In the pathological case where all 50,000 nodes have the same ID, the inner `while` loop runs 50,000 times per node, each iteration calling `generateId()` (which uses `crypto.randomUUID`). This is O(n²) with very expensive operations.

**Impact:** Import of a malicious file with all duplicate IDs would cause the browser to freeze for minutes.

**Fix:** Use a simple counter-based ID generation for duplicates instead of `generateId()` in the loop, or pre-generate a batch of unique IDs upfront.

#### L-1: `historySlice.ts:11-17` `uniqueById` is O(n²) — should be a Set

