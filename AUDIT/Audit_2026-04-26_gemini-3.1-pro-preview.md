# Аудит проекта PuuNote

## 1. Описание проекта и архитектуры
PuuNote — это концептуальный визуальный нелинейный текстовый редактор. Вместо классического сплошного текста "простыней", редактор использует структуру графа (дерева), где каждая мысль/ветка визуализируется в виде отдельной колонки, выстраивая бесконечную вложенность вправо.

**Стек технологий:** 
- React 19, TypeScript, Vite.
- Tailwind CSS для стилей, `motion/react` для анимаций интерфейса.
- `react-markdown` для рендера содержимого карточек.

**Как это работает под капотом:**
- **Flat Data Structure:** Все записи (`nodes`) хранятся в виде плоского массива объектов `PuuNode { id, parentId, order, content }`. Вычисление структуры колонок происходит на лету в `App.tsx`.
- **Локальность:** Приложение является полностью клиентским (Client-Side) и использует LocalStorage браузера (`useFileSystem` хук) для кэширования сессий, множественных заметок и "горячего" стейта.
- **Undo/Redo:** Реализован изолированный хук `useHistory`, делающий слепки состояния через глубокое сравнение (`fast-deep-equal`), что позволяет отменять практически любое действие.
- **Стэйт-менеджмент:** Управление построено на встроенном в React `Context API` (`EditorContext`), куда сверху-вниз прокидываются методы манипуляции узлами (добавление, удаление, редактирование, DND).

## 2. Удачные решения
- **Модель плоских узлов (Flat Array Tree):** Отказ от глубоких вложенных объектов `children: []` в пользу `parentId` является краеугольным правилом для удобного Drag & Drop и Undo/Redo. Это лучшее решение для данного архитектурного типа.
- **Мультиредактор (Read/Edit переключение):** Карточки элегантно меняют отображение с "только для чтения" (Markdown рендер) на `textarea` по двойному клику. Использование авто-резинового текстового поля (`AutoSizeTextarea`) оставляет UI непрерывным.
- **Политика «ленивого» контекста в истории:** Лимитирование истории (50 шагов) спасает от утечек памяти (Out of Memory) на длинных дистанциях при интенсивной работе.
- **Маршрутизация (activePath):** Концептуально правильная подсветка активного пути; визуальная подсветка ветки, на которой мы работаем, а остальное "уходит в туман".

## 3. Ошибки, узкие места и структурные неоптимальности

**3.1 Катастрофическая деградация при росте дерева $O(N^2)$ (Узкое место)**
- **Симптом:** При быстром наборе текста в карточке приложение может начать подлагивать, если в документе 300+ фрагментов.
- **Причина:** При наборе текста вызывается `setNodes` (чтобы обновить строку). На каждое изменение `nodes`, `App.tsx` запускает $O(N^2)$ алгоритм в `useMemo` для вычисления `columns` (колонки и уровни), `activePath` (построение дерева вниз/вверх) и `descendantIds` (рекурсивный обход детей). На больших документах пересобирать дерево во время `onChange` текстового поля категорически нельзя.
- **Решение:** Дебаунсить ввод (Debounce) или применять transient updates (например, Zustand) так, чтобы локальный ввод символов не триггерил пересчет всей иерархии приложения, пока юзер не "сохранит/покинет" карточку. Как альтернатива — кэшировать узлы в `Map<parentId, PuuNode[]>` для $O(1)$ выборки.

**3.2 Чрезмерный re-rendering из-за React Context**
- **Причина:** Все карточки обернуты в `React.memo`, но внутри употребляют `useContext(EditorContext)`. `EditorContext` в `App.tsx` включает `activePath` и `descendantIds`, которые пересоздаются при каждом вводе символа (см. пункт 3.1).
- **Симптом:** При изменении одного символа в одной карточке в фоне перерендериваются *совершенно все* карточки на экране (так как контекст обновился для всех консьюмеров). 
- **Решение:** Разбить контекст на Actions (функции) и ViewState. Идеально — перейти на атомарные стейт-менеджеры, не подверженные болезни Context re-renders.

**3.3 «God Object» в главном файле (App.tsx)**
- **Симптом:** `App.tsx` вырос до ~850 строк.
- **Причина:** Он обрабатывает глобальную клавиатуру (Global Keydown), Drag-n-Drop, отрисовку шапки `Header`, список файлов `Sidebar`, подвал `Footer` и логику `scrollView`.
- **Решение:** Вынести навигацию с клавиатуры в отдельный кастомный хук (`useKeyboardShortcuts`). Вынести отрисовку домена файлов/модалок в отдельный компонент `Sidebar` / `Header`.

**3.4 Уязвимость алгоритма Drag & Drop (Loops)**
- **Причина:** Пользователь может взять родительскую карточку и попытаться "перетянуть" (Drop) ее внутрь своей же дочерней карточки (сделать предка потомком самого себя).
- **Симптом:** Циклическая зависимость в `parentId`, что приведет к бесконечному циклу `while` (Infinite Loop) в алгоритме `descendantIds` и `columns`, намертво вешая вкладку браузера.
- **Решение:** При `onDrop` обязательно добавлять циклическую проверку (Cyclic Dependency Check) — нельзя назначать `targetId` родителем, если `targetId` является `descendantFrom(sourceId)`.

**3.5 Отсутствие Error Boundaries (Глобальных перехватчиков)**
- **Причина:** Нет защиты от сломанного рендера карточек. Если Markdown сломается, интерфейс схлопнется в белый экран. Ничего не будет видно.

## 4. Развитие, AI-функционал и рефакторинг

### Перед внедрением AI рекомендуется следующий план:
1. **Масштабный рефакторинг State Management:** 
   Разделить состояние на локальное (что происходит в карточке прямо сейчас, например набор текста) и глобальное (структура дерева). Интеграция библиотеки **Zustand** позволит безболезненно получать нужные узлы (через селекторы) и обновлять их без ререндера соседних колонок. Это сделает редактор сверхбыстрым, даже если AI начнет "писать (стримить)" длинный ответ в реал-тайме — UI не зависнет!

2. **Архитектурная сепарация (Free vs Paid):**
   - **Главное ядро (Core):** Оставляем бесплатным весь базовый функционал (`Card.tsx`, DND, LocalStorage, экспорт/импорт).
   - **Слой AI (Plugin Architecture):** Создать папку `src/features/ai`. AI не должен встраиваться прям в тело редактора, он должен слушать события (event-driven). Либо внедряться через HOC (Higher-Order Component) поверх оригинальной `Card`. 
   - Это гарантирует, что если у пользователя нет подписки, AI-модули просто не рендерятся и не тянут лишние зависимости/байт-коды и не перегружают UI.

### Философия применения AI в PuuNote
Данный интерфейс идеален для AI-соавторства (Co-Thinker). "Колонки" располагают к тому, чтобы AI предлагал идеи и развивал мысль в ширину:
1. **Generative Branching (Генеративное ветвление):** Пользователь нажимает на фрагменте [AI: Развей мысль] -> AI создает для этой карточки 3 новых "дочерних" узла-продолжения (соседняя колонка вправо).
2. **Contextual AI (Умный RAG):** Поскольку у нас есть `activePath` (от текущей карточки до корня), AI можно легко скармливать этот путь. Модель всегда будет знать точный контекст мысли, игнорируя соседние ветки. Это колоссальное преимущество перед плоским текстом!
3. **AI-Саммаризатор (Коллапс)**: Автоматическое схлопывание ветки из 20 дочерних элементов в одно предложение-свертку (Summerize to collapse).
