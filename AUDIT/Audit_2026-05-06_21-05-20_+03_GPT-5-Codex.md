## Что это за проект

PuuNote - браузерное React-приложение для работы с древовидными заметками. Основная идея: вместо одного длинного Markdown-файла документ хранится как набор карточек, связанных `parentId`. Корневые идеи показываются слева, дочерние раскрываются в колонках справа.

Типичный сценарий:

1. Пользователь открывает документ.
2. `useFileSystemInit()` загружает список документов и активный документ из IndexedDB.
3. Zustand-store держит текущие `nodes`, selection, UI settings, history и document actions.
4. `BoardView` строит колонки из дерева и рендерит `Card`.
5. `Card` показывает Markdown через `SafeMarkdown` или редактор (`AutoSizeTextarea` / `WysiwygEditor`).
6. Изменения идут через `documentApi` и `setNodes`.
7. `historySlice` пишет undo/redo snapshots.
8. `useFileSystem` debounce-ит сохранение в Dexie.
9. Импорт/экспорт и clipboard проходят через `markdownParser` и `documentExport`.
10. Snapshots пишутся отдельно в IndexedDB.

## Регрессия с merge-кнопкой

### Что сейчас есть

Доменная функция merge находится в `src/domain/documentTree.ts:398`. Валидация - `canMergeNodes()` в `src/domain/documentTree.ts:28`.

Store action:

- `src/store/slices/documentSlice.ts:246` - `mergeNodes(masterId, nodeIdsToMerge)`.

UI-кнопка:

- `src/components/FloatingCardActions.tsx:330` - кнопка рендерится только при `mergeValidation.ok`.
- `src/components/FloatingCardActions.tsx:56` - список merge ids собирается из `[activeId, ...selectedIds]`.

### Что я проверил

В браузере:

1. Открыл приложение.
2. Кликнул первую root-карточку.
3. Shift-click по соседней root-карточке.
4. Нажал `.` для показа card actions.
5. В accessibility snapshot появилась кнопка `Merge selected cards`.

То есть в текущем состоянии кнопка не удалена полностью.

### Почему пользователь все равно может видеть "пропала"

Merge-кнопка зависит от цепочки условий:

- есть `activeId`;
- `selectedIds` вместе с `activeId` дают минимум 2 карточки;
- все выбранные карточки существуют;
- все выбранные карточки siblings с одинаковым `parentId`;
- нет ancestor/descendant пары;
- карточка не в editing mode;
- активная карточка видна в viewport;
- floating actions сейчас видимы через hover, touch или hotkey `.`;
- `timelineOpen` выключен.

Если любое условие не выполнено, кнопки нет вообще. Не disabled, не tooltip, не toast. При invalid merge `documentSlice.ts:250` пишет только `console.warn`, пользователь ничего не видит.

### Конкретные проблемы merge UX

1. Merge существует только во floating toolbar. Нет команды в Command Palette, Header, контекстном меню или footer selection bar.
2. Invalid state не объясняется пользователю. Если выбраны не siblings, кнопка исчезает.
3. Выбор карточек не оптимизирован под merge. `Shift` между разными parents строит depth-first range и фильтрует родителей, что почти гарантированно создает selection, непригодный для merge.
4. Комментарий в `FloatingCardActions.tsx:167` говорит про Space, а реальный hotkey в `useAppHotkeys.ts` и shortcuts modal - `.`.
5. После недавнего diff подтверждение merge удалено. Сейчас маленькая floating-кнопка выполняет destructive-ish structural action сразу. Undo есть, но случайность выше.
6. Нет UI regression test на "select siblings -> merge action visible -> click -> nodes merged".

### Рекомендованное исправление merge

Минимальный надежный путь:

1. Добавить явную команду `Merge selected cards` в Command Palette.
2. Показывать merge action в footer, когда `selectedIds.length > 1`, даже если floating toolbar скрыт.
3. Если merge невозможен, показывать disabled item с reason из `canMergeNodes`.
4. Заменить `console.warn` на `toast.warning`.
5. Добавить Playwright/Vitest+RTL тест на видимость merge-кнопки и доменный результат.
6. Сделать selection для merge sibling-first: если пользователь пытается merge разных parents, давать понятный feedback.

## Найденные проблемы и решения

### P0/P1: регрессии и пользовательские риски

#### 1. Merge action слишком хрупкий и плохо обнаруживаемый

Файлы:

- `src/components/FloatingCardActions.tsx:56`
- `src/components/FloatingCardActions.tsx:330`
- `src/store/slices/documentSlice.ts:246`
- `src/domain/documentTree.ts:28`

Проблема: ключевой пользовательский action живет только во floating overlay и полностью исчезает при invalid selection. Это объясняет, почему за 4 итерации "починка" могла не сойтись: фиксили одну ветку условий, но не меняли модель UX.

Решение: вынести merge в явную команду, добавить reason/disabled state/toast, покрыть UI-регрессией.

#### 2. Autosave может иметь race между in-flight сохранениями

Файл: `src/hooks/useFileSystem.ts:242`

Сейчас debounce сбрасывает таймер, но не сериализует уже начатые `DocumentService.saveNodes()`. Если медленное старое сохранение завершится после более нового, оно потенциально может перезаписать IndexedDB старыми `nodes` и поставить `saveStatus: saved`.

Решение:

- Ввести monotonically increasing `saveVersion`.
- Сохранять только последнюю версию или сериализовать очередь.
- После resolve проверять, что `fileId` и `version` все еще актуальны.
- Добавить тест: save A зависает, save B завершается, потом A завершается и не должен перезаписать B.

#### 3. External image в tutorial делает сетевой запрос и падает в консоли

Файлы:

- `src/assets/help.md:271`
- `src/components/SafeMarkdown.tsx:40`

В browser smoke есть ошибка: `Failed to load resource: net::ERR_CONNECTION_CLOSED @ https://via.placeholder.com/150`.

Для local-first/private notes это еще и privacy issue: Markdown-картинки могут делать внешние запросы при просмотре заметки.

Решение:

- Убрать `https://via.placeholder.com/150` из tutorial, заменить локальным asset или plain example.
- Рассмотреть настройку: external images off by default, "Load external media" opt-in.
- Для публичного hosting добавить CSP, например `img-src 'self' data:` или осознанный allowlist.

#### 4. Нет E2E/regression coverage на критичные UI flows

Покрыты доменные операции, но не UI-цепочки:

- multi-select -> merge visible -> merge click;
- hover/`.` floating actions;
- edit -> autosave -> switch document;
- snapshot restore через UI;
- import large/corrupt file через UI;
- WYSIWYG round-trip.

Решение: добавить Playwright smoke suite на 5-7 ключевых потоков. Особенно merge.

### Архитектура и структура

#### 5. `Header.tsx` стал "god component"

Файл: `src/components/Header.tsx` - 1110 строк.

Он держит:

- desktop nav;
- mobile menu;
- import/export;
- theme switching;
- theme tuning;
- branch color tuning;
- fullscreen/zen;
- undo/redo;
- большое количество Zustand selectors.

Последствия:

- сложно понимать регрессии;
- сложно тестировать;
- изменение темы или nodes заставляет Header пересчитывать много UI;
- появилось дублирование `MiniSlider`.

Решение:

- Разделить на `HeaderNav`, `MobileMenu`, `ThemeTuneMenu`, `BranchColorMenu`, `ExportMenu`.
- Вынести derived hooks: `useActiveBranchColor`, `useThemeTuningControls`.
- Использовать `useShallow` для связанных store selectors.

#### 6. `src/components/MiniSlider.tsx` не используется, а в `Header.tsx` есть локальная копия

Файлы:

- `src/components/MiniSlider.tsx:19`
- `src/components/Header.tsx:58`

Это чистый хвост после итераций. Отдельный компонент даже функциональнее: умеет ввод значения по double click и `onStepDown/onStepUp`. Но Header продолжает использовать локальную версию.

Решение: импортировать `MiniSlider` из отдельного файла и удалить локальную функцию из Header. Если numeric input не нужен - удалить новый файл.

#### 7. Доменная логика дерева хорошая, но еще не имеет единого transaction layer

Файлы:

- `src/domain/documentTree.ts`
- `src/store/slices/documentSlice.ts`

Сейчас операции вроде merge, move, delete, split разбросаны между доменом и store post-effects. Это нормально для текущего размера, но регрессии вокруг selection/active/editing показывают, что нужны операции уровня "transaction result":

```ts
{
  nodes: PuuNode[];
  activeId?: string | null;
  selectedIds?: string[];
  editingId?: string | null;
  toast?: string;
}
```

Решение: постепенно перевести structural actions на `DocumentTransactionResult`, чтобы домен возвращал не только nodes, но и безопасные UI последствия.

#### 8. Tree cache глобальный и завязан на identity массива

Файл: `src/utils/tree.ts:8`

`buildTreeIndex` кеширует только последний `nodes` по ссылке. Это быстро, но хрупко:

- если кто-то мутирует массив in-place, индекс станет stale;
- один глобальный cache плохо масштабируется на несколько документов/тестов;
- sorted children пересчитываются в `orderedChildrenFromIndex()` каждый раз.

Решение:

- Использовать `WeakMap<PuuNode[], TreeIndex>`.
- Либо хранить sorted children внутри индекса.
- На dev можно `Object.freeze` нормализованные nodes, чтобы ловить мутации.

#### 9. Большие документы будут упираться в O(n) операции на каждый edit

Файлы:

- `src/domain/documentTree.ts:88`
- `src/store/slices/historySlice.ts`
- `src/components/BoardView.tsx`

Каждое изменение content делает `nodes.map`, history хранит полные snapshots, BoardView рендерит все видимые карточки.

Решение:

- Краткосрочно: лимитировать history по общему размеру, не только по числу шагов.
- Среднесрочно: хранить patches для content edits.
- Долгосрочно: нормализовать store как `nodesById + childrenIds`, а export/buildTreeIndex делать на boundaries.
- BoardView virtualization делать отдельно и осторожно.

#### 10. Selection logic слишком сложная и не совпадает с merge model

Файл: `src/store/slices/selectionSlice.ts:25`

Shift-select между siblings работает понятно. Shift-select между разными parents строит depth-first range и фильтрует родителей. Это может быть удобно для copy/cut, но плохо для merge, которому нужны только siblings.

Решение:

- Разделить selection modes: structural range/copy selection и sibling action selection.
- Для merge показывать reason: "можно объединять только карточки одного уровня".
- Упростить комментарии, убрать рассуждения из кода.

### Данные, хранение, надежность

#### 11. Dirty restore может перезаписать файл данными из sessionStorage без строгого отчета

Файл: `src/domain/documentService.ts:306`

`restoreDirtySave()` берет `puu_dirty_save`, нормализует и пишет в IndexedDB. Если sessionStorage поврежден или содержит array, который после normalize становится пустым, можно получить неожиданное пустое сохранение.

Решение:

- Использовать `normalizeNodesWithReport`.
- Если входной dirty array был непустой, а output пустой - отказаться от restore.
- Логировать/toast warning при repair.

#### 12. Snapshot restore очищает undo history

Файл: `src/db/snapshots.ts`

`restoreSnapshot()` вызывает `setNodesRaw`, который сбрасывает history. Это может быть осознанно, но для пользователя "restore" выглядит как действие, которое хочется отменить.

Решение:

- Перед restore делать текущий snapshot.
- Или restore через `setNodes` с отдельным history step.

#### 13. `saveDocuments([])` ничего не делает

Файл: `src/domain/documentService.ts:208`

Сейчас пустой список документов не сохраняется. Это не ломает текущий flow, потому что при удалении последнего документа создается новый. Но метод как API выглядит неожиданно.

Решение: либо явно документировать invariant "documents не бывает пустым", либо сделать метод честным и поддержать очистку.

#### 14. Search cache мутирует cached nodes

Файл: `src/domain/documentService.ts:358`

`cached.nodes[i].fileTitle = doc.title` мутирует объект в cache. Практически это работает, но делает cache неочевидным.

Решение: возвращать новые объекты при сборке searchNodes или хранить title отдельно.

### Security/privacy

#### 15. External media в Markdown не ограничены privacy policy

Файл: `src/components/SafeMarkdown.tsx:40`

Sanitizer режет dangerous tags и protocols, это хорошо. Но разрешенные external `img src=http/https` позволяют заметкам делать внешние запросы при просмотре.

Решение:

- По умолчанию блокировать external images.
- Показывать placeholder "external image blocked".
- Добавить настройку или per-document consent.

#### 16. Link policy не совпадает между editor и renderer

Файлы:

- `src/utils/link.ts:5`
- `src/components/SafeMarkdown.tsx:40`

WYSIWYG разрешает `mailto:`, `tel:`, relative links, anchors. `SafeMarkdown` явно оставляет только `href: ["http", "https"]`. Пользователь может создать link, который потом не рендерится как link.

Решение: унифицировать link policy. Если `mailto/tel/#/relative` нужны, добавить их в sanitizer; если не нужны, не разрешать в editor.

#### 17. AI/plugin foundation нельзя считать безопасной платформой

Файлы:

- `src/domain/aiProvider.ts`
- `src/domain/aiOperations.ts`
- `src/plugins/registry.ts`

Сейчас это mock/internal foundation. Нет manifest, permissions, sandbox, consent preview, audit log, API key policy. Это нормально для прототипа, но опасно развивать как "plugins" без границ.

Решение:

- До внешних AI providers добавить preview отправляемого context.
- Добавить explicit consent.
- Добавить provider permissions и local-only/offline mode.
- Для plugins: manifest, permissions, versioning, disable/uninstall, transaction API.

#### 18. Security headers/CSP не настроены

Файлы:

- `render.yaml`
- `vite.config.ts`
- hosting config отсутствует для headers.

Для local prototype это терпимо. Для production hosting нужны CSP, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`.

- рассуждения в `selectionSlice`.

Решение: оставить только комментарии, объясняющие invariant или сложный алгоритм. Номера итераций убрать.

#### 24. `selectionSlice` использует `as any` и disable lint

#### 28. `Header` подписан на слишком много store-полей

Файл: `src/components/Header.tsx:144`

#### 29. Homegrown Markdown-to-HTML clipboard renderer

Файл: `src/utils/markdownParser.ts:168`

Новая функция `markdownToClipboardHtml` решает полезную задачу - rich paste во внешние редакторы. Но это ручной partial parser. Он не соответствует полному GFM и может иметь edge cases.

Решение:

- Покрыть больше cases: nested lists, links with parentheses, code fences with language, tables, task lists.
- Или использовать unified/remark/rehype pipeline для HTML export.

Недостает:

### Производительность

#### 31. BoardView без virtualization

Файл: `src/components/BoardView.tsx`

Timeline виртуализирован, Board - нет. При больших документах будут лаги, потому что рендерятся все видимые колонки и карточки.

Решение: сначала собрать performance fixtures, потом добавлять virtualization только для колонок/карточек с учетом scroll alignment и DnD.

#### 32. History хранит полные массивы nodes

Файл: `src/store/slices/historySlice.ts`

Лимит 50 steps есть, но нет лимита по bytes. Для больших документов это память и GC.

Решение: size-aware history, patches для content edits, structural transactions для tree ops.

#### 33. Search index строится последовательно по всем документам

Файл: `src/domain/documentService.ts:323`

На большом количестве документов command palette может долго открываться. Cache есть, но первый запуск дорогой.

Решение:

- incremental search index per file;
- web worker для индекса;
- lazy search после открытия palette с progress.

## Как я вижу дальнейшее развитие

### 1. Стабилизация core workflows

Перед новыми функциями стоит закрепить ядро:

- карточки: add/edit/delete/split/merge/move;
- selection;
- undo/redo;
- autosave/switch document;
- import/export;
- snapshots.

Главная метрика: ключевые workflows должны иметь regression tests. Merge - первый кандидат.

### 2. Явный action model вместо скрытых floating-only действий

Floating actions удобны, но не должны быть единственным доступом к важным операциям. Нужен единый action registry:

- label;
- icon;
- shortcut;
- enabled/disabled reason;
- run;
- dangerous flag;
- placement: command palette, floating actions, footer toolbar, context menu.

Тогда merge, delete, split, AI draft, snapshot можно подключать в разные UI без дублирования условий.

### 3. Storage state machine

Сохранение лучше оформить явно:

- `idle`;
- `dirty`;
- `saving(version)`;
- `saved(version)`;
- `error(version, reason)`;
- `recoveringDirtyBackup`.

Это снимет класс race bugs и сделает save status честнее.

### 4. Отдельная линия "privacy-first"

PuuNote позиционируется как local-first. Это надо довести до продукта:

- external media disabled by default;
- explicit AI context preview;
- no silent external requests;
- export/import warnings;
- CSP;
- clear docs: где хранятся данные и что может покинуть браузер.

### 5. Умеренная AI-интеграция

AI просится в этот продукт, но только после privacy foundation:

- локальные/mock providers оставить;
- внешние providers через явный consent;
- preview context before send;
- generated nodes as drafts;
- snapshots before batch operations;
- audit log for generated content.

### 6. Большие документы

Сейчас проект хорош для малых/средних деревьев. Для больших:

- normalized store;
- patches;
- Board virtualization;
- web worker search/index;
- lazy editor loading уже частично есть, но можно улучшать дальше.

### 7. Product features, которые естественно просятся

- Selection toolbar с merge/delete/move/color/copy.
- Context menu карточки.
- Search results with path preview.
- Backlinks/internal links между карточками.
- Tags или metadata chips.
- Document templates.
- Better outline/minimap для Board.
- Snapshot diff preview.
- Import preview перед созданием документа.
- "Repair report" после импорта с подробностями.

## Общая оценка

Проект логичен по базовой архитектуре: React UI, Zustand state, domain functions, Dexie persistence, tests на дерево и import/export. Главный структурный риск сейчас - не "плохой код", а накопление быстрых feature patches без единого action/transaction слоя и без UI regression tests. Из-за этого маленькая UI-регрессия вроде исчезнувшей merge-кнопки оказывается трудноуловимой: условие находится не в одном месте, а между selection, floating overlay, validation, active card, viewport и user interaction mode.

Если закрепить merge и autosave тестами, разобрать Header и ввести общий action model, проект станет заметно устойчивее и проще для дальнейшего развития.
