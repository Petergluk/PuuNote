# Сводный аудит и roadmap проекта PuuNote

**Обновлено:** 2026-04-28
**Модель:** Gemini 3.1 Pro

Данный документ представляет собой унифицированную выжимку из всех проведенных аудитов архитектуры и кода.

## 1. Обзор проекта и архитектуры
PuuNote — это продвинутый нелинейный (древовидный) Markdown-редактор, созданный для организации мыслей, заметок и ветвящегося контента (схема Local-first).

**Стек технологий:** 
- React 18, TypeScript, Vite
- Tailwind CSS 
- Zustand (управление состоянием, undo/redo)
- React Markdown (rehype-sanitize, remark-gfm)
- Motion (анимации), Dexie.js (IndexedDB)

**Что реализовано хорошо:**
- Четкое разделение на компоненты, утилиты, обход дерева и стор данных.
- Чистый Type-check и отсутствие ESLint warning'ов.
- Разделение макета на shell (`App.tsx`) и логику построения доски (`useBoardLayout.ts`).
- Мощная система отмены/повтора (undo/redo), безопасная работа с локальным хранилищем.

## 2. Успешно устранённые проблемы (по результатам всех аудитов)

### Производительность, Безопасность и Качество кода
- **Prettier & ESLint Quality Gates:** Сборка `npm run build` проходит без предупреждений и падений. Запуск автоформаттера изолирован от контента генерации (`.prettierignore` для `AUDIT/`).
- **Race Condition при `beforeunload`:** Из-за асинхронной природы IndexedDB, таймер несохранённых данных при закрытии вкладки теперь сбрасывает данные в `localStorage` («путь экстренного грязного сохранения»), который затем подхватывается и пишется в IndexedDB при следующей инициализации БД.
- **Риск Infinite Loop:** При повреждённой структуре ID в БД введён жесткий лимит итераций (`1000`) при обходе родительских и дочерних узлов дерева в `src/utils/tree.ts`.
- **Защита XSS (Markdown):** Исключён протокол `data:` из атрибута `src` в `rehype-sanitize`. Изображения ограничены только `http` и `https`.
- **UI Блокировки:** Убраны все нативные вызовы `alert()`. Подключён встроенный механизм `openConfirm()` из Zustand Store для информирования пользователя.
- **Глобальная чистота кода:** Magic strings (например, `<!-- puunote-format: 1 -->` и `5MB`) вынесены в `src/constants.ts`.

### Визуальные исправления
- Перехват `Ctrl/Cmd+F` убран из `CommandPalette`, чтобы не нарушать нативный поиск по странице.
- Все первые верхние карточки в колонках теперь выравниваются строго по верху, а все дочерние центрируются, чтобы ветки не "плавали". Отступы скорректированы (`pt-16` вместо `pt-[50vh]` для корней дерева).

## 3. Дорожная карта (Roadmap) / Что еще предстоит сделать

Ниже перечислены структурные вещи, которые отложены на следующие этапы (или оставлены как TODO):

1. **Автоматические тесты:** В проекте нет Unit-тестов. Требуется внедрение `Vitest` для покрытия `tree.ts`, Markdown-парсера (`markdownParser.ts`) и Zod валидации. Для стабильности требуется `Playwright` smoke test.
2. **Транзакционность IndexedDB:** Создание или удаление файлов сейчас пишутся независимо в `db.documents` и `db.files`. При сбое (нехватке квоты или краше) возможен рассинхрон или сиротские записи. В будущем следует обернуть эти вызовы в единую транзакцию `db.transaction('rw', db.documents, db.files, ...)`.
3. **Глобальный поиск и индексация:** Сейчас поиск в `CommandPalette` загружает *все* текстовые узлы и перебирает их в памяти. На больших воркспейсах потребуется внедрение `fuse.js` (или полнотекстового индекса в IndexedDB).
4. **Уведомления (Toasts):** Нет единого интерфейса или всплывающих `Toast` сообщений, которые показывали бы статусы (например, «Сохранено...» или «Ошибка записи»). Использование `ConfirmDialog` для ошибок — лишь временный патч.
5. **Размер бандла:** Главный бандл достаточно нагружен. Желательно вынести `react-markdown` и `lucide-react` в раздельные чанки (`manualChunks` в Vite), чтобы ускорить первичную инициализацию на слабых устройствах.
