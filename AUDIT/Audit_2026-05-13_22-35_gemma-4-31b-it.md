# Project Audit Report: PuuNote

## 1. Project Overview

### Description
**PuuNote** is a local-first, personal knowledge management application. Unlike traditional linear note-taking apps, PuuNote employs a "branching" or "fragmented" architecture, allowing users to decompose ideas into a tree-like structure of interconnected cards (fragments). This approach facilitates non-linear thinking and structured brainstorming.

### Architecture
The project follows a clean, layered architecture that ensures a strict separation of concerns:

*   **Persistence Layer (`src/db`)**: Uses **Dexie.js** (a wrapper for IndexedDB) to store documents, node data (fragments), and snapshots locally in the browser.
*   **Domain Layer (`src/domain`)**: Contains the "brain" of the application. It handles complex business logic such as tree manipulation (`documentTree.ts`), AI-driven expansions (`aiOperations.ts`), and document exports (`documentExport.ts`).
*   **State Management Layer (`src/store`)**: Utilizes **Zustand** with a sliced pattern. It maintains the application's runtime state (UI mode, active selection, node history for undo/redo) and synchronizes it with the persistence layer.
*   **UI Layer (`src/components` & `src/hooks`)**: A React-based frontend using Tailwind CSS for styling. It implements a highly optimized "Board View" that renders the node tree across multiple columns.

### Tech Stack
*   **Frontend**: React 19, TypeScript.
*   **Styling**: Tailwind CSS 4.
*   **State Management**: Zustand 5 (with `subscribeWithSelector` for fine-grained reactivity).
*   **Database**: Dexie.js (IndexedDB).
*   **Rich Text Editing**: Tiptap (ProseMirror-based) with Markdown extensions.
*   **Search**: Fuse.js (integrated into the domain layer).
*   **OCR**: Tesseract.js.
*   **Animations**: Motion (Framer Motion).
*   **Validation**: Zod.

## 2. Code Analysis

### Logic and Robustness
The codebase is exceptionally robust. The developers have implemented several "defensive" patterns:
*   **Immutable State**: The `documentApi` consistently returns new arrays/objects, ensuring React's change detection works perfectly.
*   **Data Normalization**: The `normalizeNodes` and `normalizeNodesWithReport` functions in `DocumentService` prevent corrupted state by regenerating duplicate IDs and repairing inconsistent node orders during load/save.
*   **Debounced Persistence**: `FileSystemManager` prevents IndexedDB bottlenecks by debouncing saves (1000ms), ensuring smooth UI performance even during rapid typing.

### Performance
Performance is a clear priority in this project:
*   **Computation Memoization**: `BoardView` uses `useMemo` extensively for tree indexing and path computation, preventing expensive $O(N)$ operations on every render.
*   **Search Optimization**: `DocumentService` implements a sophisticated two-tier caching system (a global signature-based cache and a per-file cache) for search indices, drastically reducing DB reads.
*   **Resource Loading**: Heavy components (Timeline, Command Palette, Modals) are `lazy` loaded to reduce the initial bundle size.

### Structural Efficiency
The project is well-organized. The use of Zustand slices makes the store maintainable. The delegation of logic to the `domain` layer keeps components lean and focused on rendering.

### Security and Stability
*   **Local-First**: By avoiding a backend, the app eliminates most traditional network-based security vulnerabilities.
*   **Quota Management**: The app explicitly checks for `QuotaExceededError` and enforces a `MAX_FILE_SIZE_BYTES` (5MB) limit to prevent browser storage crashes.
*   **Error Boundaries**: Top-level UI components are wrapped in `ErrorBoundary` components, preventing a single component failure from crashing the entire application.

### "Tails" and Dead Code
A search for `TODO`, `FIXME`, and `console.log` revealed almost no incomplete features or debugging remnants in the source code (most logs were restricted to test files).

## 3. Problem Categorization

The audit found no **Critical** issues. The identified points are **Non-Critical** architectural refinements.

### Non-Critical Issues

#### A. Overloaded Initialization Hook
*   **Problem**: `useFileSystemInit` in `src/hooks/useFileSystem.ts` is a "god hook" that handles migration, document listing, active file resolution, node hydration, and state synchronization.
*   **Risk**: Low. It is currently functional, but as the app grows, this hook will become a maintenance bottleneck and difficult to test.
*   **Fix**: Extract the logic into a dedicated `InitializationService` or a set of smaller, focused hooks (e.g., `useDbMigration`, `useDocumentHydration`).
*   **Risk of Fix**: Very low; purely a refactoring effort.

#### B. Simple Title Heuristic
*   **Problem**: `deriveDocumentTitle` uses a basic heuristic (first meaningful line of the first root node).
*   **Risk**: Low. Users might find the auto-generated titles inaccurate for complex documents.
*   **Fix**: Implement a dedicated `title` field in `PuuDocumentMetadata` that can be manually overridden by the user.
*   **Risk of Fix**: Low; requires a small schema update and a UI element for renaming.

#### C. Linear Search Indexing
*   **Problem**: `getSearchNodes` performs a linear scan of all documents to check for updates.
*   **Risk**: Low/Medium. With hundreds of documents, this could cause a noticeable lag during the first search after a restart.
*   **Fix**: Implement a more robust indexing strategy (e.g., a separate `search_index` table in Dexie) rather than relying on the `updatedAt` timestamp of the metadata.
*   **Risk of Fix**: Medium; requires changing how search data is persisted and updated.
