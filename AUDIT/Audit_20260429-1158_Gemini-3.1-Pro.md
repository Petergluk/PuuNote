# Аудит проекта PuuNote

**Дата:** 2026-04-29 11:58  
**Модель:** Gemini 3.1 Pro

## 1. Обзор проекта и архитектура

**PuuNote** — это локальный (Local-first) нелинейный Markdown-редактор, где заметки (фракталы/фрагменты) организуются в древовидные структуры. Проект находится в активной фазе подготовки к интеграции AI-плагинов.

**Стек технологий:**
- **Фреймворк:** React 19, TypeScript, Vite
- **Стилизация:** Tailwind CSS v4
- **Управление состоянием:** Zustand
- **Хранилище:** Dexie.js (оболочка над IndexedDB)
- **Иконки & Анимации:** lucide-react, motion (framer-motion)

**Как это работает:**
- Пользователь работает с `PuuNode` (узлами), каждый из которых содержит Markdown-текст и ссылку `parentId`. 
- Интерфейс выстраивает родительско-дочерние отношения в горизонтальные колонки, либо показывает ленту (TimelineView).
- База данных через систему снимков (snapshots) и локальное автосохранение защищает данные.
- Присутствует заложенный фундамент для LLM: `JobRunner` (инструмент для управления фоновыми потоками и статусами AI) и `contextExtraction` (для сборки дерева в текстовый промпт).

## 2. Найденные проблемы, уязвимости и узкие места

В ходе аудита было выявлено несколько логических ошибок, способных привести к порче данных или "незаметным" багам:

### 2.1 Критические & Утечки Данных (Data Loss)
1. **Утечка последних изменений в `AutoSizeTextarea.tsx`**
   - *Проблема:* Компонент использует Debounce (задержку 500мс) перед применением изменений. Но при событии "unmount" (размонтирование карточки при смене вида, удалении соседа или клике по другой доске) `useEffect` cleanup просто сбрасывает таймер `clearTimeout`. Текущий текст в `pendingChangeRef` ─ исчезает навсегда.
   - *Решение:* В cleanup блоке `useEffect` необходимо добавлять `if (pendingChangeRef.current) onChange(pendingChangeRef.current)`.

2. **Порча Markdown цитат при импорте (`markdownParser.ts`)**
   - *Проблема:* В функции `parseMindMapFormat` применяется агрессивный regex `n.content.replace(/^\s*>[ \t]?/gm, "")`. Это означает, что при импорте или вставке формата mindmap система **жестко удалит** все блок-квоты (`> моя цитата`) из всех строк текста карточки.
   - *Решение:* Заменить логику: удалять маркеры цитат только с первой строки узла (если экспортёр добавляет их) или вообще убрать это зачищение, чтобы не разрушать пользовательский контент.

3. **Опасное слияние узлов (`mergeNodes` в `documentTree.ts`)**
   - *Проблема:* Метод позволяет слить массив выделенных узлов (`nodeIdsToMerge`) в один (`masterId`). При этом он не проверяет, являются ли некоторые выделенные узлы родителями/детьми друг друга. Слияние перепривяжет их потомков на `masterId`, но сами дети будут удалены из графа (так как они в массиве на слияние). Это приведет к дублированию текста при потере иерархии (орфанные узлы).
   - *Решение:* Исключать из массива слияния те узлы, которые являются потомками других выделенных узлов, либо запрещать `mergeNodes`, если есть такие отношения.

### 2.2 Архитектура, Оптимизация и Память
4. **Тяжелый Undo/Redo (`useAppStore.ts`)**
   - *Проблема:* При каждом изменении буквы, в историю откатов (`past`) копируется **весь** массив объектов графа. Для крупных проектов это O(N) по памяти на каждый чих, что сожрет RAM и вызовет фризы из-за Garbage Collector.
   - *Решение:* Заменить слепое копирование на патчи (например, `zundo` или `immer` патчи). Либо сделать сохранение истории для Undo/Redo потранзакционным (только измененные узлы через Snapshot, а text-editing вынести из глобального undo).

5. **Мутация кэшированной мапы в обходе дерева (`tree.ts`)**
   - *Проблема:* При получении детей `childrenMap.get(parentId)` возвращается массив по ссылке. Затем код вызывает `children.sort(...)`. Это мутирует исходный массив прямо в структуре `TreeIndex`, что может привести к багам кэширования при перерисовках.
   - *Решение:* Сортировать только копию: `[...children].sort(...)`.

6. **Временная сложность сборщика промпта (`contextExtraction.ts`)**
   - *Проблема:* Строка 74 и далее собирает отображение дерева. `allDfNodes` вызывается по `getDepthFirstNodes`, а затем фильтруется. В больших документах глубокий обход на каждый запрос LLM будет "тормозить" основной поток UI (он тяжелый и синхронный).
   - *Решение:* Использовать `useMemo` для предрасчета глубин дерева, либо инкапсулировать `depth` в исходный массив узлов.

### 2.3 UX, Мусор и Хвосты
7. **Мусорный файл:** Корневой пустой файл `/untitled.tsx`.
8. **Range Selection Stub (`useAppStore.ts`):** Оставлен незаконченный хвост (todo) для выделения через Shift: `// Let's just do simple toggle for now...`. Из-за этого массовое выделение диапазона через Shift не работает.
9. **Запутанная логика UI Mode:** Кнопка выхода из Zen Modal сбрасывает `uiMode` и принудительно вызывает `exitFullscreen`. А слушатель браузерного full_screen сбрасывает `uiMode` в `normal`. Могут возникнуть гонки состояний, если пользователь использует инструменты ОС (а не API браузера).

## 4. Как я вижу дальнейшее развитие проекта?

Документ движется в сторону системы "Графовых ИИ-агентов". Для этого нужно произвести следующие оптимизации и нововведения:

1. **Реальный Streaming UI компонентов:**
   Вместо того чтобы просто блокировать карточку (`isGenerating`), лучше показывать текст прямо во время потокового ответа из LLM, используя `ReadableStream`.
2. **Менеджмент медиа (Images & Files):**
   Нужно интегрировать работу с файлами (картинки, записи). `IndexedDB` (на Dexie) отлично поддерживает `Blob`. Добавить конвертацию drag&drop файлов в Blob БД и генерацию `URL.createObjectURL(blob)`, чтобы пользователи могли вставлять скриншоты.
3. **CRDT для коллаборации и мульти-окон:**
   Перенести стейт на Yjs. Даже без сети (в single-player) Yjs позволит открыть доску в двух разных вкладках браузера и они будут синхронно безшовно изменяться друг об друга без потерь. Кроме того, Yjs автоматически даст мощное и легкое Undo/Redo "из коробки" без костылей с массивами из Zustand.
4. **Command Layout Palette (Макродизайн доски):**
   Пользователям сложно сразу ориентироваться. Нужна кнопка-шаблон "Премиксы" доски ("Сделать как расписание на неделю", "Сделать канбан", "Пустое дерево").
5. **Теги и Фильтры (Graph Query):**
   Добавить `#tags` в Metadata. С помощью Search Bar позволить скрывать все колонки/карточки, не относящиеся к поиску, сохраняя древовидную прослеживаемость.
