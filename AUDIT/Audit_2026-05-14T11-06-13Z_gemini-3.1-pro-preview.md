**Аудитор:** Gemini 3.1 Pro (AI Studio Coding Agent)
**Дата:** 2026-05-14

---

## 1. Описание проекта

**PuuNote** — это продвинутый текстовый редактор с древовидной структурой (outliner / бесконечная доска). Проект позволяет создавать так называемые «карточки» (фрагменты), которые выстраиваются в многоуровневое дерево. Проект имеет фокус на ведении заметок, структурировании идей и использовании AI для преобразования контента.

### Основной стек и архитектура:
- **Окружение:** React 19, Vite, TypeScript.
- **Хранение данных:** Локальное в браузере (IndexedDB) через библиотеку `dexie`. Приложение рассчитано на работу оффлайн (Local-first).
- **Стейт-менеджмент:** `zustand`. Стейт разбит на слайсы (History, Document, UI, Selection). 
- **Пользовательский интерфейс:** Tailwind CSS для стилизации, `framer-motion` для анимаций, `lucide-react` для иконок.
- **Редактор текста:** Поддержка Markdown, визуальный редактор на базе `Tiptap` + обычный текстовый Markdown-редактор (`react-textarea-autosize`).
- **Плагинная система:** Собственный реестр плагинов (`src/plugins/registry.ts`), который позволяет добавлять хуки, кнопки в карточки и команды.
- **Функционал AI:** 
  1. Существует заглушка для генерации дочерних карточек (`mockAiProvider`).
  2. Реализован плагин `voice-fixer` для записи звука с микрофона, транскрибации аудио и последующей интеллектуальной обработки/очистки текста через Google Gemini API.

## 2. Найденные проблемы и узкие места

Ниже представлен детализированный аудит кодовой базы, разделенный по уровню критичности.

### 🔴 Критичные проблемы (Рекомендуется исправить немедленно)

#### 1. Утечка API-ключей в клиентском коде / Небезопасное хранение
**Где:** `src/plugins/voice-fixer/utils.ts`
**Проблема:** Приложение читает `.env` переменные (`VITE_GLOBAL_GEMINI_API_KEY`) и выполняет REST-запросы к Gemini напрямую из браузера (`generativelanguage.googleapis.com`). Если проект будет задеплоен и доступен публично, API-ключ попадет в клиентский bundle. Любой пользователь сможет достать этот ключ из вкладки Network или исходного кода и израсходовать лимиты/средства. 
**Решение:** 
- Если это сугубо локальное приложение (Local-first), ключи должны запрашиваться у пользователя через интерфейс и сохраняться строго в его `localStorage`.
- Если это публичный SaaS, все AI-запросы ОБЯЗАНЫ проходить через собственный backend (Node.js/Express), который будет безопасно хранить ключ и общаться с API Gemini.

#### 2. Катастрофическая утечка памяти при работе с историей (Zustand Undo/Redo)
**Где:** `src/store/slices/historySlice.ts`
**Проблема:** Механизм Undo/Redo хранит историю из 50 шагов. На каждом шаге делается массивный клон *всего* состояния узлов (`[...state.past.slice(...), currentNodes]`). Так как `nodes` — это массив всех абзацев и карточек документа, на большом документе (сотни узлов) 50 глубоких/поверхностных копий массива приведут к фризам UI и быстрому переполнению памяти вкладки (вылет с *Out of Memory*).
Функция `diffAndEmit` также сканирует весь массив карточек с алгоритмической сложностью O(N).
**Решение:** Для реализации Undo/Redo в сложных древовидных структурах необходимо использовать патчи. Рекомендуется внедрить `immer` и плагин для Zustand (`zustand/middleware/immer` или `zundo`), которые под капотом хранят только разницу (diff state patch), а не копию всего массива объектов.

#### 3. Отсутствие реального AI провайдера для базовой функции "Draft Child Cards"
**Где:** `src/domain/aiProvider.ts`, `src/domain/aiOperations.ts`
**Проблема:** В интерфейсе есть функция «AI Draft Child Cards» (помощь в раскрытии карточки), которая генерирует дочерние идеи. Но она привязана к `mockAiProvider`, который выдает случайный зашитый текст-заглушку. 
**Решение:** Необходимо внедрить реального AI-провайдера, используя `@google/genai` TypeScript SDK (наподобие того, как реализована генерация через XHR в плагинеVoice Fixer).

### 🟡 Некритичные проблемы / Структурные неоптимальности (Технический долг)

#### 4. Неэффективный ререндеринг и пересчет древовидного индекса
**Где:** `src/components/BoardView.tsx`, `src/utils/tree.ts`
**Проблема:** Метод `buildTreeIndex(nodes)` вызывается внутри `useMemo` при *каждом* изменении массива `nodes`. Но так как `updateContent` (набор текста) меняет массив `nodes`, полное дерево сбрасывается и пересчитывается при *каждом нажатии клавиши*. Это bottleneck (бутылочное горлышко) для производительности на средних документах.
**Решение:** Разделить состояние контента узла и архитектуру дерева. 
- *Вариант А:* Оптимизация на уровне хранилища (Entity Adapter паттерн) – хранить мапу `ids: [], entities: {}`.
- *Вариант B:* Обновлять контент через ссылки или отдельным дебаунс-стейтом, чтобы `updateContent` не вызывал рескан всей доски до момента потери фокуса (blur).

#### 5. Ручная реализация XHR вместо использования нативного fetch
**Где:** `src/plugins/voice-fixer/utils.ts`
**Проблема:** Разработчик написал громоздкую обертку `xhrFetch` через устаревший API `XMLHttpRequest`. Это создает мусорный код в проекте, который использует React 19 и собирается Vite.
**Решение:** Заменить обертку на стандартный `fetch()` + `async/await`, либо использовать официальный SDK для Gemini (`@google/genai`). SDK сам позаботится о ретраях, правильных заголовках и извлечении данных.

#### 6. Ломающий сплит (Разделение карточки по курсору)
**Где:** `src/components/Card.tsx` (функция `handleSplitNode`)
**Проблема:** Кнопка «Split node» в сыром Markdown (`AutoSizeTextarea`) режет текст тупо по индексу каретки (`selectionStart`). Если курсор внутри HTML-тега, списка или \`**жирного текста**\`, форматирование сломается в обеих новых карточках.
**Решение:** Написать более умную функцию разделения, учитывающую перенос форматирующих Markdown-символов (AST Analysis), либо отключать эту функцию для Markdown-режима, если курсор находится внутри сложной синтаксической конструкции.

#### 7. Тяжелое управление CSS переменными в `style={{}}`
**Где:** `src/components/Card.tsx`
**Проблема:** `branchStyle` вычисляется динамически и прокидывается в проп `style={{...}}`. Рект не всегда хорошо оптимизирует объекты с 10+ custom properties.
**Решение:** Это не страшно при небольшом количестве карточек, но можно вынести логику в Tailwind `theme()` генератор, если потребуется оптимизация.

#### 8. Хранение аудиозаписи в оперативной памяти (Base64 Blob)
**Где:** `src/plugins/voice-fixer/utils.ts`
**Проблема:** Audio File считывается через `FileReader` в DataURL. Для длинного диктофона (1-2 часа) конвертация аудиопотока в гигантскую Base64 строку гарантированно повесит вкладку браузера на несколько секунд.
**Решение:** В будущем – использовать чанки или отправлять файл как `FormData`/binary payload, когда будет реализован backend прокси (вместо прямой отсылки inlineData в Gemini).

- [ ] **[Оптимизация]** Дебаунсить (`debounce`) вычисление индекса дерева `buildTreeIndex`, либо вынести изменение текста узла во внутреннее локальное состояние Компонента <Card>, которое будет отправлять данные в глобальный Store только по `onBlur`.
