# PuuNote - Project Audit

## 1. Project Overview & Architecture
PuuNote is an advanced, non-linear (tree-based) markdown editor designed for organizing thoughts, notes, and branching content.

**Tech Stack:** 
- React 18
- TypeScript
- Vite
- Tailwind CSS (with specific custom utility variables)
- Zustand (State management)
- React Markdown (plus rehype-sanitize, remark-gfm for rendering)
- Motion/react (for animations/transitions)

**Architecture:**
- **State (`src/store/useAppStore.ts`)**: Centralized local-first data model holding `nodes` (cards), `documents`, history (`past`/`future`), UI state.
- **Components (`src/components/`)**:
  - `Header`, `Footer`: Standard layout wrappers.
  - `Card`: The primary unit of content. Contains editable textarea and read-only markdown rendering.
  - `TimelineView`: Alternative hierarchical "branch" view.
  - `CommandPalette`: Keyboard-accessible action menu.
  - `FileMenu`: Drawer for switching between top-level documents.
  - `ConfirmDialog`, `ErrorBoundary`, `FullScreenModal`, `ShortcutsModal`.
- **Utils (`src/utils/`)**:
  - Functions for ID generation, tree traversals, parsing/exporting markdown recursively, and validating JSON schema.
- **Hooks (`src/hooks/`)**:
  - Managing file system (IndexedDB/localStorage) and general user preferences (shortcuts).

**What is Good:**
- Excellent modularity and clear file structure.
- State management leverages Zustand cleanly.
- Implements robust undo/redo for branching changes.
- Safe markdown parsing that sanitizes dangerous inputs while supporting GitHub-flavored markdown.
- Keyboard shortcuts system is comprehensive.

## 2. Identified Issues & Proposed Solutions

### 2.1 Code Quality & TypeScript
- **Issue**: There were several `'any'` typings assigned to `document` and `document.documentElement` to bypass TypeScript errors for fullscreen APIs.
- **Solution**: The typings have been strictly defined for vendor-prefixed properties like `webkitFullscreenElement`. An unused variable warning and an empty `catch` block in `App.tsx` were also addressed.

### 2.2 UI Consistency (Icons & Action Buttons)
- **Issue**: The group of top-left interaction buttons (files, view toggle, search palette) had mismatched visual states (some yellow, some white, different spacing, different hover borders) which led to structural inconsistency.
- **Solution**: We unified the classes to ensure identical `p-1.5 rounded-lg text-app-text-muted hover:text-app-text-primary hover:bg-app-card-hover border border-transparent hover:border-app-border` styling, and properly conditionally styled active states with `text-app-text-primary bg-app-card-hover border border-app-border`.

### 2.3 Potential Vulnerabilities
- **Data Persistence Strategy**: Currently, the system persists data likely to LocalStorage or IndexedDB without encryption. This is acceptable for a local-first editor, but a potential risk if exported backups fall into the wrong hands. In standard deployments, this is standard, but worth documenting for privacy.
- **Markdown Sandbox**: XSS vectors are mostly mitigated via `rehype-sanitize`. However, allowing custom iframe classes or `mailto` links could technically still lead to minor phishing issues. The schema whitelist appears solid.

### 2.4 Structural Weaknesses
- **Monolithic `App.tsx`**: `App.tsx` holds a lot of column-calculating logic in `useMemo` hooks (100+ lines).
  - *Recommendation*: Extract the column computation, scrolling ref logic, and fullscreen listeners into separate custom hooks (e.g., `useColumns`, `useActivePathScroll`). This would dramatically clean up `App.tsx`.
- **Zustand Store Payload**: `useAppStore.ts` stores *both* application domain data (nodes, documents) and minor UI state (colWidth, timelineOpen, uiMode).
  - *Recommendation*: Divide the store into `useDataStore` (for nodes, past, future) and `useUIStore` (for flags like timelineOpen, theme, fileMenuOpen). It simplifies debugging and prevents unnecessary rerenders on unrelated components.

## 3. Future Development & Evolution
- **Collaboration Mode / Sync**: Implement a cloud/Firebase backend syncing engine. Local-first is great, but users will eventually want cross-device usage.
- **Refactoring App.tsx**: Decouple the complex column rendering logic into a standalone component (`<BoardView />` or `<NodeColumns />`), parallel to how `<TimelineView />` is separated.
- **Drag & Drop**: Continue polishing DND. The structural math for node insertion is robust, but the visual cues could be spruced up using `framer-motion` or `pragmatic-drag-and-drop`.
- **Search Optimization**: The `CommandPalette` handles action-searching well, but a full-text global search for node contents across *all* files/branches would be a massive usability win.
- **Plugins/Extensions**: Creating a defined API payload allows the editor to add features like "Export to PDF", "Integrate with Notion", or "Web Clippers".
