# PuuNote - Аудит проекта

## 1. Обзор проекта и архитектура
PuuNote — это продвинутый нелинейный (древовидный) Markdown-редактор, созданный для организации мыслей, заметок и ветвящегося контента.

**Стек технологий:** 
- React 18
- TypeScript
- Vite
- Tailwind CSS (со специфическими кастомными переменными)
- Zustand (управление состоянием)
- React Markdown (включая rehype-sanitize, remark-gfm для рендеринга)
- Motion/react (для анимаций и переходов)

**Архитектура:**
- **Состояние (`src/store/useAppStore.ts`)**: Централизованная локальная модель данных. Хранит узлы (`nodes` - карточки), документы (`documents`), историю (`past`/`future`) и состояние интерфейса.
- **Компоненты (`src/components/`)**:
  - `Header`, `Footer`: Стандартные обертки макета.
  - `Card`: Основная единица контента. Содержит редактируемое текстовое поле и режим чтения (рендер Markdown).
  - `TimelineView`: Альтернативный иерархический вид ("линейный/ветвящийся").
  - `CommandPalette`: Меню действий с доступом с клавиатуры.
  - `FileMenu`: Выдвижная панель для переключения между документами верхнего уровня.
  - `ConfirmDialog`, `ErrorBoundary`, `FullScreenModal`, `ShortcutsModal`.
- **Утилиты (`src/utils/`)**:
  - Функции для генерации ID, обхода дерева, рекурсивного парсинга/экспорта Markdown и валидации JSON-схемы.
- **Хуки (`src/hooks/`)**:
  - Управление файловой системой (IndexedDB/localStorage) и общими настройками пользователя (горячие клавиши).

**Что реализовано хорошо:**
- Отличная модульность и понятная структура файлов.
- Управление состоянием с помощью Zustand реализовано чисто и логично.
- Мощная и надежная система отмены/повтора (undo/redo) для древовидных изменений.
- Безопасный парсинг Markdown, который санирует опасный ввод и при этом поддерживает GitHub-flavored markdown.
- Продумана и грамотно внедрена система горячих клавиш.

## 2. Найденные проблемы и предложенные решения

### 2.1 Качество кода и TypeScript
- [x] **Проблема**: В коде присутствовало несколько типов `'any'`, назначенных объектам `document` и `document.documentElement`, чтобы обойти ошибки TypeScript при использовании API полноэкранного режима (fullscreen).
  - *Решение*: Жестко переопределены для нестандартных свойств с вендорными префиксами.
- [x] **Проблема (Новая)**: Проваливался `npm run typecheck` из-за неиспользуемого `p2` в 3 местах с чекбоксами (Card, TimelineView, FullScreenModal).
  - *Решение*: Вынес `/src/utils/markdownParser.ts` как предлагал аудит (или просто переименовал в `_p2`), и `typecheck` теперь проходит.
- [x] **Проблема (Новая)**: Проваливался код-стайл `npx prettier --check`.
  - *Решение*: Выполнен `prettier --write` и добавлены команды `format:check` в `package.json`. Сборка теперь начинается с `tsc --noEmit`.

### 2.2 Визуальная консистентность (Иконки и кнопки действий)
- [x] **Проблема**: Группа кнопок взаимодействия в левом верхнем углу (файлы, переключатель вида, поиск) имела несогласованные визуальные состояния.
  - *Решение*: Унифицированы классы.

### 2.3 Потенциальные уязвимости
- **Стратегия хранения данных**: В настоящее время система сохраняет данные в локальное хранилище в открытом виде (без шифрования). Это нормально для local-first редактора, но потенциально рискованно, если к устройству есть доступ у третьих лиц. Стоит задокументировать это в политике приватности или настройках.
- [x] **Песочница Markdown**: Векторы XSS-атак в основном надежно предотвращаются библиотекой `rehype-sanitize`. Однако было технически возможно использовать сомнительные схемы в ссылках (например `mailto:`, `irc:`) и абсолютные пути для картинок.
  - *Решение*: Ограничены разрешенные протоколы для атрибутов `href` (`http`, `https`, `mailto`) и `src` (`http`, `https`, `data`) во избежание возможных векторов атак.

### 2.4 Структурные слабости приложения
- [x] **Монолитный `App.tsx`**: Файл `App.tsx` содержал слишком много логики вычисления колонок в хуках `useMemo` (более 100 строк).
  - *Решение*: Вынесено в `useColumns` и `useActivePathScroll` в `src/hooks/useBoardLayout.ts`.
- [ ] **Перегруженный Zustand-стор**: `useAppStore.ts` хранит *одновременно* и предметную область приложения (nodes, documents), и мелкие состояния интерфейса (colWidth, timelineOpen, uiMode).
  - *Рекомендация*: В будущем разделить стор на `useDataStore` и `useUIStore`.
- [x] **Неиспользуемые зависимости**: Были оставлены `@tanstack/react-virtual`, `clsx`, `tailwind-merge`. Старое название проекта `puunote-ai` в `package.json`.
  - *Решение*: Зависимости и утилиты удалены, название проекта изменено på `puunote`.
- [x] **Потенциальная потеря данных при Refresh/закрытии (Race condition)**: В хранении через `IndexedDB` мог возникнуть race condition если таймер не сбросился и вкладка закрылась.
  - *Решение*: Исправлена логика `pendingSave`, добавлен `window.addEventListener('beforeunload')` с гарантированным сохранением `flushPendingSave()` перед закрытием.

### 2.5 Нерешённые проблемы логики и надежности (Высокий/Средний приоритет)
- [x] **Целостность дерева и защита от циклов**: Валидация Zod на данный момент проверяет только структуру объекта (id, content, parentId), но не проверяет граф на наличие циклов (cycles), сирот (orphans) или отсутствующих родителей. 
  - *Решение*: В `schema.ts` добавлены лимиты (`max(50_000, ...)`), а также обход дерева с выявлением циклов и выбросом ошибки. Функции `computeActivePath`, `getDepthFirstNodes`, `moveNode` и экспорт теперь используют `visited` сеты для предотвращения зависания парсера при разрушенном графе БД.
- [x] **Дублирование логики (DRY)**: Логика обработки кликов по чекбоксам (`handleToggleCheckbox`) задублирована в трех местах: `Card.tsx`, `TimelineView.tsx`, `FullScreenModal.tsx`. 
  - *Решение*: Вынесена общая утилита `toggleCheckboxContent` в `src/utils/markdownParser.ts`.
- [x] **Слепые Fallback-и в хранилище данных**: В `useFileSystem.ts` функция `switchFile` при ошибке чтения из IndexedDB слепо подставляет данные `INITIAL_NODES` и переключает на них UI. Это опасно, так как последующий автосейв перезапишет реальный поврежденный документ стартовыми данными.
  - *Решение*: Ошибка чтения теперь явно фиксируется переменной `didFail` внутри функции `useFileSystem.ts`. Если документ не может быть загружен и не является `default`, пользователю показывается `alert`, а процесс переключения прерывается.
- [x] **Оптимизация TimelineView**: `TimelineView` вручную реализует обход дерева (depth-first), хотя в `tree.ts` уже есть готовая утилита `getDepthFirstNodes`.
  - *Решение*: Реализация в `TimelineView` заменена на вызов `getDepthFirstNodes(nodes)`, обходя лишние Map-аллокации.
- [x] **Неиспользуемая функция**: Ошибка TypeScript и предупреждения на неиспользуемые импорты `/src/utils/tree.ts` (`getAncestors`).
  - *Решение*: Удалено.
- [ ] **Отсутствие тестов (Unit/E2E)**: Проект завязан на сложную логику персистентности, парсинга Markdown и мутаций дерева, но в нем полностью отсутствуют автоматические тесты (Vitest/Playwright).
- [ ] **Неоптимальный глобальный поиск**: В `CommandPalette` поиск читает абсолютно все файлы из `IndexedDB` (`db.files.toArray()`) на каждое нажатие (даже с debounce). На больших базах это приведет к зависаниям. Необходимо внедрить поисковый индекс (например, `fuse.js` или подобное).
- [x] **Hardcoded тексты и модалки**: В `Footer.tsx` жестко зашито название туториала `"PuuNote: Complete Guide"`. В целом присутствует смешение английского и русского языков без строгой конфигурации пространств имен i18n, плюс туториал сбрасывается с потерей истории состояния.
  - *Решение*: Название добавлено в `constants.ts` -> `TUTORIAL_DOCUMENT_TITLE`. Сама модалка подтверждения из компонента `Footer.tsx` была вынесена в отдельный компонент `TutorialModal.tsx` для чистоты архитектуры.
- [ ] **Silent Errors и отсутствие индикаторов статуса сохранения**: Нет глобальной системы тостов или визуального статуса `Saving... / Saved`, а операции создания/удаления документов отражают изменения в Zustand `store` до того, как `IndexedDB` подтвердит успешную транзакцию.

## 3. Дальнейшее развитие и эволюция проекта
- **Совместная работа (Collaboration Mode) / Синхронизация**: Внедрение облачного бэкенда для синхронизации. Подход Local-first — это замечательно, но в какой-то момент пользователям потребуется бесшовная работа/синхронизация между разными устройствами.
- **Рефакторинг `App.tsx`**: Отделить сложную логику рендеринга колонок в автономный компонент (`<BoardView />` или `<NodeColumns />`), аналогично тому, как сейчас вынесен компонент `<TimelineView />`.
- **Drag & Drop**: Продолжать улучшать перетаскивание (DND). Математика структуры для вставки узлов надежна, но визуальную обратную связь (анимации перехвата, плавность) можно значительно улучшить с помощью `framer-motion` или библиотек наподобие `@atlaskit/pragmatic-drag-and-drop`.
- **Оптимизация поиска**: Палитра `CommandPalette` хорошо справляется с действиями, но глобальный полнотекстовый поиск по содержимому узлов *во всех* документах и ветках стал бы огромным плюсом к юзабилити.
- **Плагины и Расширения**: Создание определенного API/формата данных для редактора позволит добавлять сторонние фичи (например, "Экспорт в PDF", "Интеграция с Notion" или веб-клипперы для браузеров).
