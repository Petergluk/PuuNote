# Audit Report: PuuNote
**Date:** 2026-05-01
**Model:** gemini-pro-latest

## 1. Описание проекта (Архитектура, Структура, Механика)
PuuNote — это иерархическое / многоколоночное приложение для ведения заметок, структурно напоминающее Obsidian Canvas или Gingko App. 

**Стек технологий:**
- **React 19** + **Vite**
- **Zustand** — стейт-менеджмент (разбитый на слайсы: `documentSlice`, `historySlice`, `selectionSlice`, `uiSlice`).
- **Tailwind CSS** + часть UI-паттернов для стилизации.
- **Dexie.js / IndexedDB** — локальное хранилище данных (offline-first архитектура, с сохранением слепков / снапшотов).
- **React Markdown / rehype-sanitize** — для безопасного рендеринга сложного форматирования (Markdown).
- **ИИ-ассистент** (`aiOperations.ts`, `aiProvider.ts`) для генерации, обобщения или изменения контента.
- Мультиязычность через **i18next**.

**Механика работы:** 
Пользователь управляет деревом нод, которые отображаются в "Колонках" (дочерние элементы открываются правее родителей, образуя ветки). При клике на карточку фокус сдвигается по колонкам (за это отвечает хук `useBoardLayout`). Также есть `TimelineView` для плоского просмотра хронологии/версий.

---

## 2. Удачные решения
1. **Offline-first база на Dexie**: Вменяемо спроектированная изолированная база данных (`src/db/db.ts` и `snapshots.ts`), позволяющая приложению работать без сети и сохранять сложный стейт.
2. **Изоляция слайсов Zustand**: Структура стора грамотно разделена на домены (`selectionSlice.ts`, `documentSlice.ts` и т.д.), что сильно снижает риск путаницы состояний в большом приложении.
3. **Безопасный рендеринг разметки**: Разнесено редактирование текста (`AutoSizeTextarea`) и показ готовой разметки (`SafeMarkdown`), что защищает от XSS метками `rehype-sanitize`.
4. **Легковесные AI-адаптеры**: В `domain/` вынесена логика работы ИИ, не перемешивая ее жестко с UI-компонентами.

## 3. Найденные проблемы в коде (и возможные решения)

| Проблема | Пояснение | Возможное решение (которое можно отложить!) |
| --- | --- | --- |
| **"Гонки" в скролле (requestAnimationFrame)** | Логика скролла в `useBoardLayout.ts` написана через 2 `requestAnimationFrame` подряд. Если стейт дергается, фокус начинает "прыгать". Это частая причина "регрессий" (когда мы исправляем дерганья, но ломаем плавность инициализации). | Заменить ручную математику с `getBoundingClientRect()` на нативные `scrollIntoView({ behavior: 'smooth', block: 'nearest' })` по React Ref'ам. |
| **Хрупкий захват элементов** | В приложении активно используется `document.getElementById('card-...')` и `col.querySelectorAll`, что ломает "React-way" и делает UI уязвимым перед любым изменением верстки. | Хранить мапы `refs` прямо в списке карточек и обращаться к ним напрямую. |
| **Сложная логика Shift-Select** | Механизм выделения `selectionSlice.ts:24` содержит сложную эвристику (*"предотвращает выделение родителя, если его дочерние тоже выделены"*). Это дает непредсказуемый результат для пользователя. | Перейти к классической модели: выделяется просто путь до предка, либо плоский массив нод от точки А до точки Б, без исключений. |
| **TS ошибки `any`** | Линтер ругался на использование `any` в `AutoSizeTextarea` и `selectionSlice`. | **Уже исправлено** нами только что. |
| **Дублирование данных / Паника линтера** | Ранее в проекте болталась папка `temp_puunote` с запасными файлами, которая сводила ESLint с ума (выпало 132 ошибки `tsconfigRootDir`). | **Уже удалена.** Теперь линтер снова чист. |

## 4. Ошибки, неудобства и глюки UI & UX

1. **Глюк с "проваливанием" первой карточки при загрузке** 
   - **Симптом:** Тот самый глюк: *"при загрузке первая карточка оказывается в самом низу"*.
   - **Причина:** Верстка в `BoardView.tsx` содержит псевдо-отступы `pt-[95vh]` наверху колонок (вероятно, чтобы позволить карточке находиться по центру экрана). При первой загрузке математика `offsetTop` в хуке срабатывает так, что скроллит колонку не туда из-за огромного `padding-top`.
   - **Решение:** (Если захотите починить) — заменить `pt-[95vh]` на более вменяемые `pt-[30vh]` или сделать отдельный обработчик пустого пространства над картами (Spacer div).

2. **Запутанность на глубоких уровнях вложенности**
   - **Симптом:** При ширине свыше 4 колонок теряется визуальный "путь" фокуса.
   - **Возможное решение:** Сильнее "димить" (делать блеклыми) карточки, которые не входят в "Активный Путь" (`activeAncestorPath`).

3. **Коллизии горячих клавиш (UX)**
   - **Симптом:** При редактировании больших текстов, если пользователь случайно использует навигационные хоткеи.
   - **Возможное решение:** Точно убедиться, что внутри `textarea` слушатели событий глобального стора (например на удаление или свайп узлов) заглушены.
