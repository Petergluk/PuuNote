## Что это за проект

PuuNote - локальный браузерный редактор нелинейных заметок. Основная модель данных - дерево карточек. Пользователь пишет не в одном вертикальном документе, а в наборе карточек, связанных отношениями parent-child. Корневые карточки находятся в первой колонке, их дети - во второй, дальше дерево раскрывается вправо.

Текущий продуктовый образ:

- local-first Markdown editor;
- горизонтальное дерево карточек;
- inline editing;
- keyboard-first навигация;
- Markdown rendering с sanitization;
- импорт/экспорт Markdown;
- несколько локальных документов;
- IndexedDB persistence;
- TimelineView для линейного чтения;
- command palette с поиском;
- первые AI/plugin-oriented инфраструктурные слои.

### P1. `switchFile` все еще не валидирует сохраненные nodes

Файл: `src/hooks/useFileSystem.ts`

Initial load вызывает `validateNodes(savedNodes)`, но `switchFile` берет `saved` из IndexedDB и сразу reduce/dedupe.

Проблема:

- если документ в IndexedDB поврежден, initial active file может быть защищен, но при переключении на другой документ validation не применяется;
- cycles/depth/orphans/invalid metadata могут попасть в UI.

Решение:

- создать единый `normalizeNodes(raw: unknown): PuuNode[]`;
- использовать его в initial load, switchFile, import, dirtySave restore, snapshot restore.

### P1. Ошибка сохранения нового документа не останавливает state update


То есть UI может показать документ, которого нет в IndexedDB.

Решение:

### P1. Snapshot restore не привязан к active document consistency

`restoreSnapshot(snapshotId)` просто делает `setNodesRaw(snapshot.nodes)` для текущего active store. Он не проверяет, что snapshot относится к текущему active document, и не переключает `activeFileId`.

Риск:

- snapshot одного документа можно восстановить поверх другого;
- если в UI появится список snapshots разных документов, это станет data-loss bug.

Решение:

- проверять `snapshot.documentId === activeFileId`;
- или переключать active document на `snapshot.documentId`;
- перед restore делать snapshot текущего состояния;
- показывать confirm.

### P2. DOM Virtualization внедрена частично и может создавать новые UX-глюки

Виртуализация сейчас добавлена только в `TimelineView`, но подключена не полностью согласованно с scroll-моделью приложения.

Файл: `src/components/TimelineView.tsx`

Сейчас используется:

```tsx
<Virtuoso
  useWindowScroll
  data={orderedNodes}
  itemContent={...}
/>
```

Но приложение устроено как full-height shell: верхний контейнер имеет `h-screen overflow-hidden`, а вертикальный скролл Timeline происходит внутри `#main-scroller`, где при `timelineOpen` включается `overflow-y-auto`. То есть `useWindowScroll` слушает window, хотя реальный scroll container - внутренний DOM-элемент.

Вероятные симптомы:

- Timeline может не дорендеривать элементы при прокрутке;
- jumpy scroll;
- неверная высота virtual list;
- scroll-to-heading работает нестабильно;
- active node может не попадать в viewport;
- поведение отличается между desktop/mobile из-за разных scroll containers.

Решение:

- убрать `useWindowScroll`;
- либо дать Virtuoso собственный scroll container с фиксированной высотой;
- либо получить `#main-scroller` ref и передать его как `customScrollParent`;
- держать scroll ownership в одном месте, а не смешивать `main-scroller`, `window` и `scrollIntoView`.

### P2. Outline navigation несовместима с virtualized DOM

Файл: `src/components/TimelineView.tsx`

`scrollToNode` сейчас делает:

```ts
const element = document.getElementById(`tl-node-${nodeId}`);
if (element) {
  element.scrollIntoView({ behavior: "smooth", block: "start" });
  setActiveId(nodeId);
}
```

При virtualization элементы вне viewport физически отсутствуют в DOM. Поэтому клик по heading в outline для далекого node может ничего не сделать: `getElementById` вернет `null`.

Решение:

- создать `const virtuosoRef = useRef<VirtuosoHandle>(null)`;
- построить `nodeId -> index` map по `orderedNodes`;
- в outline использовать `virtuosoRef.current?.scrollToIndex({ index, align: "start", behavior: "smooth" })`;
- после scroll выставлять `activeId`.

### P2. Active scroll logic для Timeline фактически не работает

Файл: `src/hooks/useBoardLayout.ts`

`useActivePathScroll` всегда сначала ищет board-card:

```ts
const activeEl = document.getElementById(`card-${activeId}`);
if (!activeEl) return;
if (timelineOpen) {
  activeEl.scrollIntoView({ behavior: "smooth", block: "nearest" });
  return;
}
```

В Timeline элементы имеют id `tl-node-${id}`, а не `card-${id}`. Поэтому при `timelineOpen` код чаще всего выходит на `if (!activeEl) return` и не делает timeline scroll вообще.

Решение:

- не использовать board-layout hook для timeline;
- вынести отдельный `useTimelineActiveScroll`;
- или передавать в Timeline `activeId` и скроллить через Virtuoso API;
- убрать timeline branch из `useActivePathScroll`, чтобы не было ложного ощущения поддержки.

### P2. Virtuoso items не имеют явного `computeItemKey`

Файл: `src/components/TimelineView.tsx`

Сейчас `key={n.id}` находится внутри rendered item. Для virtual list лучше явно задать stable key на уровне Virtuoso:

```tsx
computeItemKey={(_, node) => node.id}
```

Без этого при reorder/move/merge/active editing возможны мелкие glitches:

- reuse DOM не для того node;
- потеря фокуса;
- textarea state прыгает;
- неверные измерения высоты после editing.

Решение:

- добавить `computeItemKey`;
- проверить editing active item при move/merge/switch timeline.

### P2. Virtualization снизила DOM cost, но не CPU cost

`TimelineView` виртуализирует DOM-рендер, но каждый раз строит:

- `orderedNodes = getDepthFirstNodes(nodes)`;
- `outline` проходом по всем ordered nodes и всем строкам content.

Это нормально для небольших документов, но при больших документах каждый edit меняет `nodes`, а значит пересчитывает весь depth-first list и outline.

Решение:

- мемоизировать tree index выше;
- считать outline лениво или кэшировать per document version;
- debounce outline rebuild при наборе текста;
- для очень больших документов строить outline из heading index.

### P2. Fuzzy search стал функциональнее, но не масштабируемее

### P2. Context extraction для LLM без budget и privacy controls

Файл: `src/domain/contextExtraction.ts`

`buildContextForLLM` может включить всех descendants без ограничения (`maxLevels = -1`), не считает tokens/chars, не фильтрует sensitive metadata, не ограничивает размер node content.

Риски:

- случайная отправка слишком большого контекста;
- дорогие/медленные AI-запросы;

Решение:

- `maxChars` / `maxTokens`;
- include/exclude ancestors/descendants/siblings options;
- preview перед отправкой;
- redaction hooks;
- return structured context + warnings, а не только text blob.

Проблемы:

- registry singleton;
- no lifecycle integration with React;
- `console.log` in library path;
- hooks не вызываются из document operations;
- card actions typed with `React.ReactNode` without importing React type;
- plugin actions have direct `onClick(nodeId)` with no permission/error boundary.

Решение:

- определить plugin lifecycle;
- вызывать hooks из единого document service;
- заменить logs на dev-only logger;
- обернуть plugin callbacks в try/catch;
- добавить permissions/capabilities хотя бы на уровне типов.

### P2. Job cancellation semantics misleading

Файл: `src/domain/jobRunner.ts`

`cancelJob(jobId)` только добавляет id в set. Если job function не вызывает `checkCancelled`, отмены не будет. UI тоже пока не показывает cancel.

Решение:

- документировать cooperative cancellation;
- добавить AbortController в job API;
- `runJob(name, ({ signal, updateProgress }) => ...)`;
- добавить status `"cancelled"` вместо `"failed"` для отмены.

### P2. Snapshot storage может быстро раздувать IndexedDB

Каждый snapshot сохраняет полный `PuuNode[]`. Нет лимитов, TTL, retention policy, compression или delta snapshots.

Решение:

- хранить максимум N snapshots на документ;
- auto-delete старые;
- показать размер storage;
- рассмотреть compression/diff later.

### P2. `updatedAt` все еще string в DB и number в app

`PuuDocument.updatedAt` - number, `DocumentMeta.updatedAt` - string.

Результат:

- постоянные conversions;
- больше шансов на sort/date bugs.

Решение:

- Dexie v3 migration к numeric `updatedAt`;
- убрать casts.

### P2. `validateNodes` дедуплицирует last occurrence, но не нормализует order

Сейчас duplicate IDs схлопываются через `Map`, сохраняя last occurrence. Но:

- order может иметь gaps/duplicates;
- siblings order не нормализуется;
- metadata не ограничена;
- пустой content валиден, но empty document handling спорный.

Решение:

- сделать `normalizeDocumentNodes`;
- per-parent order normalization;
- metadata schema;
- clear return type with warnings.

### P2. Empty document persistence все еще спорная

Autosave пропускает `nodes.length === 0`.

Если продуктово empty document невозможен - это надо жестко гарантировать. Если возможен - надо сохранять empty array.

Решение:

- выбрать invariant;
- добавить тест;
- убрать silent skip или enforce root node.

### P2. Import/export format не lossless

PuuNote Markdown markers могут конфликтовать с пользовательским HTML comment:

- `<!-- puunote-node -->`
- `<!-- puunote-format: 1 -->`

Generic Markdown import по-прежнему эвристический.

Решение:

- добавить JSON export для lossless backup;
- экранировать internal markers;
- использовать Markdown AST для generic import;
- добавить golden roundtrip tests.

### P2. README переоценивает security

README говорит `Zero XSS risks`. Это слишком сильное заявление.

Да, `rehype-sanitize` снижает риск, но:

- Markdown images can load remote URLs;
- dependencies can have future vulnerabilities;
- browser rendering/security is not mathematically zero-risk;
- plugin/AI layer может добавить новые surfaces.

Решение:

- заменить на "XSS risk is reduced by sanitization and protocol restrictions".

### P3. Store все еще крупный и browser-dependent

Да, mutations вынесены в `documentApi`, но store все еще содержит:

- export blob/download;
- filename generation;
- confirm dialog;
- UI flags;
- selection;
- history;
- document mutation wrappers.

Решение:

- split slices: `documentSlice`, `uiSlice`, `historySlice`, `selectionSlice`;
- export service вынести из store;
- browser-only code держать в UI/service layer.

### P3. Tests есть, но покрытие пока минимальное

Сейчас тестируются:

- split;
- delete;
- move child;
- merge simple case;
- tree index/path/descendants/depth-first/cycle.

Не покрыто:

- import/export roundtrip;
- validation/normalization;
- switchFile;
- persistence errors;
- snapshots;
- selection/merge edge cases;
- command search;
- context extraction;
- checkbox toggling;
- AI/job cancellation.

Решение:

- расширить unit tests вокруг domain + storage boundaries;
- добавить e2e smoke с Playwright позже.

### P3. Generated-state UI завязан на `metadata.isGenerating`

Карточка показывает shimmer при `node.metadata?.isGenerating`, но нет clear ownership:

- кто ставит/снимает флаг;
- что при failed job;
- должен ли flag сохраняться в IndexedDB;
- очищать ли generating states при app startup.

Решение:

- transient UI job state держать в job store, а не persist metadata;
- если нужно сохранять, добавить recovery cleanup on init.

### P3. Code style issues указывают на незавершенность этапа

Примеры:

- `{" "}` в JSX;
- длинные className strings;
- inconsistent quote style in new code;
- comments типа "improve later";
- console logs в plugin registry;
- `catch(e) {}`.

Решение:

- format;
- cleanup pass;
- small component extraction;
- remove stale comments.

## Финальная оценка

Текущий этап полезный, но сырой. Архитектурно проект движется в правильную сторону: появились domain layer, tests, jobs, plugins, snapshots и LLM context. Но эти части пока не собраны в надежную систему. Перед добавлением реальных AI-фич лучше сделать stabilization pass: привести качество репозитория к green checks, убрать несоответствия документации, закрыть persistence validation gaps и формализовать public API.

После этого проект будет готов к аккуратному внедрению AI-функций без риска превратить UI-компоненты в место, где живут prompt logic, storage mutations, snapshots и side effects одновременно.
