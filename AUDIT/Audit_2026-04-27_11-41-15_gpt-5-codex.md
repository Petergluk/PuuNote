## Что это за проект

PuuNote - React/Vite SPA для локального ведения заметок в виде дерева карточек. Вместо одной линейной страницы документ разбивается на узлы. Узлы отображаются колонками слева направо: корневые карточки слева, дети правее, следующие уровни еще правее.

Основные сущности:

- `PuuNode` (`src/types.ts`): `id`, `content`, `parentId`, `order`.
- `PuuDocument` (`src/types.ts`): `id`, `title`, `updatedAt`.

Модель дерева плоская: все карточки лежат в одном массиве `nodes`, связь строится через `parentId`, порядок соседей через `order`. Это удачная и простая модель для хранения, импорта, экспорта и миграций.

## Найденные проблемы и решения

### 1. Lint сейчас падает: ref мутируется во время render

Файл: `src/App.tsx:77-78`

```ts
const colRefs = useRef<(HTMLDivElement | null)[]>([]);
colRefs.current.length = columns.length;
```

Это нарушение React rules: refs не должны читаться/мутироваться во время render, если это не безопасная lazy initialization. ESLint `react-hooks/refs` уже падает на этом.

Возможные последствия:

- поведение может быть нестабильным в StrictMode/Concurrent rendering;
- CI будет красным;
- "исправление" очистки refs само стало регрессией.

Решение:

- убрать `colRefs.current.length = columns.length` из render;
- чистить refs в effect:

```ts
useEffect(() => {
  colRefs.current = colRefs.current.slice(0, columns.length);
}, [columns.length]);
```

- или отказаться от массива refs и использовать callback refs с Map/WeakMap.

### 2. Lint падает на `prefer-const`

Файл: `src/hooks/useFileSystem.ts:123`, `src/hooks/useFileSystem.ts:259`

Переменные `savedNodes` и `saved` не переassign'ятся.

Решение:

- заменить `let savedNodes = fileData?.nodes` на `const savedNodes = fileData?.nodes`;
- заменить `let saved = fileData?.nodes` на `const saved = fileData?.nodes`.

### 3. `npm audit` все еще красный из-за неиспользуемого PWA пакета

Файлы:

- `package.json:50`
- `vite.config.ts:11-32`

`VitePWA` закомментирован, но `vite-plugin-pwa` остался в `devDependencies`. В итоге vulnerability все еще присутствует в lockfile и ломает audit:

`vite-plugin-pwa -> workbox-build -> @rollup/plugin-terser -> serialize-javascript`.

Решение:

- если PWA пока не используется, удалить `vite-plugin-pwa` из `devDependencies` и обновить lockfile;
- если PWA нужен, обновить до версии, где закрыта цепочка, или использовать overrides только после проверки совместимости;
- удалить закомментированный `VitePWA`-блок из `vite.config.ts`, если feature отложена.

### 4. Build проходит, но bundle слишком большой

Текущий JS chunk: 779.29 KB minified / 243.03 KB gzip.

Вероятные причины:

- `react-markdown`, `remark-gfm`, `rehype-sanitize`;
- `motion`;
- `dexie`;
- `zod`;
- `i18next` + language detector + react integration;
- command palette и несколько больших UI режимов в одном entry chunk.

Решение:

- lazy-load `TimelineView`, `FullScreenModal`, `CommandPalette`;
- вынести Markdown renderer в отдельный dynamic chunk;
- рассмотреть легкий i18n вместо `i18next` для нескольких строк;
- добавить bundle analyzer;
- manual chunks для `markdown`, `motion`, `storage/vendor`.

### 8. `computeActivePath` все еще логически спорный

Файл: `src/store/useAppStore.ts:57-89`

Функция строит путь от root до active node, а потом автоматически продолжает путь вниз по первому ребенку. Это делает "active path" не только выбранным путем пользователя, а еще и произвольной descendant веткой.

Проблема:

- подсветка может показывать путь, который пользователь не выбирал;
- `Footer` depth считает не только ancestor depth, но и auto-descendant;
- поведение зависит от первого ребенка по `order`.

Решение:

- сделать `activePath` только ancestors + active node;
- если нужна "прогнозируемая ветка вниз", хранить отдельное понятие `focusedBranch`/`selectedPath`;
- переименовать функцию, если текущее поведение намеренное.

Проблема:

- сортировка, обход и parent/children mapping могут расходиться;
- сложнее оптимизировать;
- сложнее тестировать.

Решение:

- добавить `src/utils/tree.ts`;
- вынести `buildTreeIndex`, `getOrderedChildren`, `getDepthFirstNodes`, `getAncestors`, `getDescendants`, `normalizeOrders`;
- покрыть эти функции unit-тестами.

### 10. Markdown parser все еще путает PuuNote-format и обычный Markdown

Файл: `src/utils/markdownParser.ts:34-39`

```ts
const isPuuNoteFormat =
  mdText.includes("<!-- puunote-format: 1 -->") ||
  /(?:^|\n)[^\S\n]*---[^\S\n]*(?:\n|$)/.test(mdText);
```

Любой обычный Markdown с `---` будет считаться PuuNote-format. Это ломает:

- YAML front matter;
- horizontal rules;
- документы, где `---` используется как разделитель.

Решение:

- считать PuuNote-format только при marker `<!-- puunote-format: 1 -->`;
- legacy `---`-format поддерживать отдельной кнопкой/режимом "Import old PuuNote format";
- добавить тесты на YAML front matter, horizontal rules, fenced code blocks.

### 11. Markdown export не экранирует собственный delimiter

Файл: `src/utils/markdownParser.ts:4-31`

Export использует строки `---` как разделители узлов. Если content карточки содержит standalone `---`, round-trip может разбиться на лишние nodes.

Решение:

- хранить структуру в JSON/metadata, а Markdown только как content;
- или экранировать delimiter внутри content;
- или использовать HTML comments с node metadata вместо plain `---`.

### 12. `parseMindMapFormat` все еще слишком эвристический

Файл: `src/utils/markdownParser.ts:106-180`

Parser смешивает heading-level и indentation/list-level. Он не использует Markdown AST, поэтому edge cases будут ломаться:

- nested lists с разной шириной indent;
- code blocks;
- blockquotes;
- tables;
- YAML front matter;
- horizontal rules.

Решение:

- парсить через `remark` AST;
- явно определить mapping headings/lists -> nodes;
- добавить round-trip/import fixtures.

### 13. `validateNodes` проверяет shape, но не проверяет граф

Файл: `src/utils/schema.ts`

Zod проверяет поля одного node, но не проверяет:

- уникальность id;
- parentId ссылается на существующий node;
- отсутствие циклов;
- корректность sibling `order`;
- отсутствие orphan nodes.

Часть уникальности id решается при загрузке, но графовая целостность не гарантируется.

Решение:

- добавить `normalizeAndValidateTree(nodes)`;
- удалять/ремонтировать orphan nodes;
- разрывать циклы;
- пересчитывать `order` внутри siblings;
- возвращать diagnostics пользователю при импорте.

### 14. Миграция IndexedDB удаляет legacy localStorage даже при частичном провале

Файл: `src/hooks/useFileSystem.ts:36-74`

Код ловит ошибку миграции, но после `catch` все равно удаляет:

- `puu_documents`;
- `scribe_nodes`.

Также внутри цикла `db.files.put(...).catch(() => {})` подавляет ошибки записи конкретного документа.

Риск: при частичной ошибке IndexedDB старые данные могут быть удалены из legacy storage до успешной миграции.

Решение:

- мигрировать в транзакции Dexie;
- удалять legacy keys только после полного успеха;
- хранить `migrationCompleted` flag;
- не подавлять ошибки `db.files.put`.

### 15. `flushPendingSave` не гарантирует завершение записи

Файл: `src/hooks/useFileSystem.ts:14-26`

`flushPendingSave` запускает `db.files.put(...).catch(...)`, но не возвращает Promise и callers его не await'ят. Перед `switchFile` это лучше, чем просто потерять timer, но запись все равно асинхронна и может завершиться позже или упасть.

Решение:

- сделать `flushPendingSave(): Promise<void>`;
- `await flushPendingSave()` в `switchFile`, `deleteFile`, `createNewFile`;
- обрабатывать ошибку сохранения через UI state/toast.

### 16. `pendingSave` глобальный singleton вне React/store

Файл: `src/hooks/useFileSystem.ts:8-12`

Состояние pending save хранится module-level объектом. Это работает для одного mounted app, но плохо тестируется и может вести себя неожиданно при hot reload/StrictMode/multiple roots.

Решение:

- вынести persistence в отдельный service/class с явным lifecycle;
- или хранить pending save внутри hook через refs и возвращать actions через context/store;
- добавить тесты на switch/delete/create во время pending save.

### 17. Удаление документа без подтверждения

Файлы:

- `src/components/FileMenu.tsx:81-85`;
- `src/components/CommandPalette.tsx:156-161`;
- `src/hooks/useFileSystem.ts:315-334`.

Удаление документа происходит сразу. Для редактора заметок это высокий UX-риск.

Решение:

- confirm dialog;
- undo delete document;
- soft delete/trash;
- auto-backup перед delete.

### 18. `deleteFile` вызывает `switchFile` без `await`

Файл: `src/hooks/useFileSystem.ts:327-328`

`switchFile` async, но вызов внутри `deleteFile` не await'ится. Потом `db.files.delete(fileId)` и `db.documents.delete(fileId)` идут параллельно. Скорее всего это не ломает happy path, но порядок side effects неявный.

Решение:

- `await switchFile(newDocs[0].id)`;
- сделать все file actions возвращающими Promise;
- в UI показывать pending state.

### 19. FullScreen "Focus Mode" все еще не соответствует названию

Файл: `src/components/FullScreenModal.tsx:39-44`

При открытии `nodeId` код берет `targetNode`, а затем показывает все `columnNodes` с тем же `parentId`. То есть это не focus одного node, а column/siblings editor.

Решение:

- либо переименовать UI в "Column Focus";
- либо показывать только выбранный node и, возможно, его subtree/context.

### 20. Search ищет только по текущему документу и без debounce

Файл: `src/components/CommandPalette.tsx:41-43`

Для маленького документа нормально, но при больших trees фильтрация всех nodes на каждый ввод может быть заметной. Также command palette не ищет по documents.

Решение:

- debounce query;
- индексировать lowercase content;
- поиск по всем документам через IndexedDB;
- показывать путь/документ результата.

### 21. Command palette hotkey открывает, но не toggle

Файл: `src/components/CommandPalette.tsx:24-33`

`Cmd/Ctrl+K` всегда делает `setIsOpen(true)`. Escape закрывает только когда focus внутри input. Это приемлемо, но может быть неожиданно: повторный `Cmd+K` не закрывает palette.

Решение:

- `setCommandPaletteOpen(!current)` или отдельный `toggleCommandPalette`;
- глобальный Escape handler для закрытия overlay.

### 22. `canUndo`/`canRedo` как функции в store ухудшают реактивность UI

Файл: `src/store/useAppStore.ts:213-214`, `src/components/Header.tsx:26-27`

Header подписывается на функцию `canUndo`, а не на `past.length`. Функция стабильна, но компонент все равно может ререндериться из-за других selected slices. Сейчас, вероятно, работает, но модель не идеальна.

Решение:

- хранить derived booleans через selectors: `useAppStore((s) => s.past.length > 0)`;
- убрать `canUndo`/`canRedo` actions.

### 23. Import через `setNodesRaw` не создает новый документ

Файл: `src/App.tsx:152-176`

Импорт `.md` заменяет nodes текущего active document и сбрасывает history. Возможно это ожидаемо, но для пользователя опасно: импорт может перезаписать текущий документ.

Решение:

- при import создавать новый документ;
- или показывать confirm: replace current document / create new;
- перед import делать backup snapshot.

### 24. Footer reset tutorial тоже использует `setNodesRaw`

Файл: `src/components/Footer.tsx:65-71`

Кнопка `?` заменяет текущий документ на tutorial и сбрасывает history. Tooltip говорит "Use Ctrl+Z to undo", но `setNodesRaw` очищает history, поэтому это может быть неправдой.

Решение:

- для reset tutorial использовать `setNodes`, если действие должно быть undoable;
- или поменять tooltip и добавить confirm.

### 25. `localStorage` writes не обернуты в try/catch

Файлы:

- `src/hooks/usePreferences.ts`;
- `src/hooks/useFileSystem.ts`.

IndexedDB ошибки частично ловятся, но `localStorage.setItem` может бросить `QuotaExceededError`/security exception.

Решение:

- storage adapter с safe get/set/remove;
- UI warning при невозможности сохранить preferences/active id.

### 26. Privacy формулировки слишком сильные

README/help говорят, что "мы ничего не отправляем" и "absolute data privacy". Но Markdown external images и links могут обращаться в сеть, browser extensions могут читать IndexedDB/localStorage, а dev server слушает `0.0.0.0`.

Решение:

- переформулировать на "local-first";
- добавить настройку block external media;
- описать privacy boundaries.

### 27. Dev server открыт на `0.0.0.0`

Файл: `package.json:7`

`vite --host=0.0.0.0` удобно для контейнеров, но локально открывает dev server в сеть.

Решение:

- `dev`: `vite --port=3000`;
- `dev:host`: `vite --port=3000 --host=0.0.0.0`.

### 28. i18n внедрен частично

Файл: `src/i18n.ts`

Переведены только несколько строк. Большая часть UI остается на английском/русском вперемешку: Footer, Timeline copy, FullScreen labels, tooltips, README/help.

Решение:

- либо убрать i18next и оставить один язык;

- добавить namespace/resources JSON вместо inline object.

### 29. `clsx`/`tailwind-merge` helper почти не используется

Файл: `src/lib/utils.ts`

Есть `cn`, но компоненты в основном используют ручные template strings. Это не критично, но dependency/helper сейчас почти декоративный.

Решение:

- начать использовать `cn` в компонентах;
- или удалить `clsx`/`tailwind-merge` до появления design-system компонента.

### 30. Нет тестов

В проекте нет test runner и тестов. При текущей логике это риск: storage migration, undo/redo, Markdown import/export и drag/drop легко ломаются незаметно.

Решение:

- Vitest для utils/store/parser;
- Playwright для базовых UI сценариев;
- fixtures для Markdown import/export;
- тест на migration localStorage -> IndexedDB.

## Потенциальные уязвимости

1. `npm audit` high vulnerabilities через `vite-plugin-pwa`.  
   Даже build-time vulnerability может ломать CI/deploy и supply-chain policies.

2. External Markdown media.  
   `react-markdown` может отрендерить `img` с внешним URL, браузер сделает сетевой запрос.

3. Хранилище не защищено от XSS/расширений.  
   IndexedDB/localStorage local-first, но не encrypted/secure storage.

4. Import DoS частично ограничен, но не полностью.  
   Есть file size 5 MB и per-node content 5 MB, но количество nodes не ограничено, graph не валидируется.

5. Dev server на `0.0.0.0`.  
   Может быть нежелательно в локальной сети.
